channel

channel的内部实质就是一个缓冲池+两个协程队列（send recv）如果 Channel 是有缓冲的，缓冲区（buf）的内存也会在堆上分配。

分配内存

Channel 是一个引用类型。在 Go 中，Channel 是一个引用类型，它的底层数据结构（hchan）是在堆上分配的。当创建一个 Channel 时，Go 运行时会调用 mallocgc 函数在堆上分配内存。

  • Channel 是 Goroutine 之间通信的核心机制，需要在多个 Goroutine 之间共享。
  • 堆上的内存可以被所有 Goroutine 访问，而栈上的内存是 Goroutine 私有的。

注意这里的设计思想：

  • 在 Channel 的发送和接收操作中，会先尝试快速路径，即在不加锁的情况下检查是否可以立即完成操作。
  • Channel 的发送队列（sendq）和接收队列（recvq）是基于无锁队列实现的。这些队列使用原子操作（如 CAS）来管理 Goroutine 的入队和出队，避免了锁的开销。
  • Channel 使用条件变量（如 gopark 和 goready）来实现 Goroutine 的挂起和唤醒。这些机制避免了显式的锁竞争，提高了并发性能。

阻塞与非阻塞

nil Channel：向 nil Channel 发送或接收数据会永久阻塞（除非是非阻塞模式）。【nil Channel 是一个未初始化的 Channel 变量，其值为 nil。比如：var ch chan int // ch 是一个 nil Channel】

在 Go 语言中，阻塞模式和非阻塞模式是指 Goroutine 在执行 Channel 操作时的行为方式。这两种模式的区别在于，当 Channel 操作不能立即完成时，Goroutine 是否会挂起等待。

阻塞模式

在阻塞模式下，如果 Channel 操作不能立即完成（例如向无缓冲 Channel 发送数据时没有接收方，或向有缓冲 Channel 发送数据时缓冲区已满），当前 Goroutine 会被挂起，直到操作可以继续。

  • Goroutine 会等待，直到 Channel 操作完成。
  • 这是 Go 中 Channel 的默认行为。

如果 ch 是无缓冲 Channel，ch <- 1 会阻塞，直到有其他 Goroutine 执行 <-ch。

如果 ch 是有缓冲 Channel，ch <- 1 只有在缓冲区满时才会阻塞。

非阻塞模式

在非阻塞模式下，如果 Channel 操作不能立即完成，Goroutine 不会挂起，而是直接返回失败（例如返回 false）。

  • Goroutine 不会等待，而是立即返回。
  • 非阻塞模式通常通过 select 语句的 default 分支实现。

如果 ch 的缓冲区已满，ch <- 1 会立即执行 default 分支。如果 ch 的缓冲区为空，<-ch 会立即执行 default 分支。

异步有阻塞

有缓冲 Channel（c.dataqsiz > 0）。缓冲区已满（c.qcount == c.dataqsiz）。没有 Goroutine 在 recvq 队列中等待接收数据。这时发送操作会阻塞，当前 Goroutine 会被挂起，直到缓冲区有空闲空间。

异步无阻塞

有缓冲 Channel（c.dataqsiz > 0）。缓冲区未满（c.qcount < c.dataqsiz）。没有 Goroutine 在 recvq 队列中等待接收数据。发送操作会直接将数据写入缓冲区，不会阻塞。并立即返回成功。

同步无阻塞

无缓冲 Channel（c.dataqsiz == 0）。有 Goroutine 在 recvq 队列中等待接收数据。如果有等待接收的 Goroutine，发送操作会直接调用 send 函数，将数据拷贝到接收方的目标地址，并唤醒接收方的 Goroutine。

同步阻塞

无缓冲 Channel（c.dataqsiz == 0）。没有 Goroutine 在 recvq 队列中等待接收数据。发送操作会阻塞，当前 Goroutine 会被挂起，直到有接收方接收数据。等待接收方唤醒。

关闭

chan close之后，所有阻塞的recvq 和 sendq（recvq和sendq只有有一个队列存在）中的sudog，清除sudog的一些数据和状态，设置 gp.param = nil， 让上层逻辑知道这是因为 close chan导致的。唤醒所有的g之后，g就会 继续执行 chansend 或者 chanrecv 中剩余的逻辑，也就是释放sudog（这也就是为什么 closechan 不需要释放sudog的原因）

传统 IPC 机制的对比：

特性 Channel（Go） 传统 IPC
通信模型 共享内存 + 通信同步 共享内存 + 显式同步
遵循 CSP 模型，通过通信共享内存 依赖于共享内存、消息队列、信号量等机制
并发模型 Goroutine（轻量级） 进程（重量级）
M:N 调度（Goroutine 调度到多个线程） 1:1 调度（每个进程对应一个线程）
通信方式 双向通信（通过不同 Channel 实现发送和接收） 单向或双向通信（双向通信需要额外机制）
支持阻塞和非阻塞操作（通过 select 实现非阻塞） 通常是阻塞的，非阻塞操作需要额外配置
类型安全，只能传递特定类型的数据 无类型安全，需自行处理序列化和反序列化
同步机制 隐式同步（发送和接收自动阻塞，直到对方准备好） 显式同步（需使用锁、信号量等机制）
无锁设计（通过原子操作和条件变量实现高效同步） 锁竞争可能导致性能问题
性能 高效（Goroutine 创建和切换开销小） 较低效（进程创建和切换开销大）
低延迟（适合高并发场景） 高延迟（通信开销较大）
使用场景 Goroutine 间通信（高并发应用，如 Web 服务器） 进程间通信（分布式系统、多进程应用）
