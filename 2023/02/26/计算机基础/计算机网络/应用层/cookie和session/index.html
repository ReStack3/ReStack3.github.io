身份验证和会话管理

一文彻底搞清session、cookie、token的区别 - 知乎

在 Web 开发中，session、cookie 和 token 都是用于维护客户端与服务器之间的状态和认证信息的技术。它们有不同的功能、存储方式和使用场景，下面是它们的具体解释及生活中的例子。

Session（会话）

定义：

Session 是服务器端存储的临时数据，它通常用于记录用户的登录状态、权限、偏好设置等信息。每次用户访问一个网站时，服务器会为该用户创建一个唯一的会话，称为会话 ID。该会话 ID 会通过 cookie 或 URL 传递给客户端。

特点：

  • 存储位置：服务器端。
  • 生命周期：会话通常会在一段时间内保持活跃，或者在用户退出时被销毁。服务器会根据该 ID 找到对应的会话数据。
  • 安全性：因为数据存储在服务器上，所以相对安全，只有服务器能访问用户的会话数据。

生活中的例子：

可以将 session 比作酒店的 房间卡。当你到达酒店时，前台为你提供一张房间卡，它记录了你的房间号和一些信息。你可以凭借房间卡随时进入房间。虽然房间卡在你手上，但房间数据存储在前台（服务器），只有你和酒店管理人员知道房间数据。

常见使用：

  • 用户登录时，服务器创建一个 session 来保存用户的登录状态。
  • 如果你关闭浏览器，session 通常会过期或被销毁。

Cookie（小数据存储）

定义：

Cookie 是存储在客户端浏览器中的小型数据。每次客户端发起请求时，浏览器会自动将相关的 cookie 数据发送给服务器。cookie 主要用于保存用户的偏好设置、身份认证信息等。

特点：

  • 存储位置：客户端浏览器。
  • 生命周期：cookie 可以设置过期时间，若没有设置，浏览器关闭时会自动删除。
  • 大小限制：一般每个 cookie 的大小有限（通常是 4KB），并且同一网站可以存储多个 cookie。

生活中的例子：

可以将 cookie 比作 超市会员卡。你在超市注册会员时，超市会给你一张卡片，卡片上记录了你的会员信息（如积分）。每次你去超市购物时，会员卡会自动被扫描，超市系统能自动识别你的身份并记录相关信息。

常见使用：

  • 记录用户的登录状态。
  • 存储用户的语言选择、主题偏好等。
  • 用于跟踪用户的浏览行为（广告投放等）。

Token（令牌）

定义：

Token 是一种用于身份认证的字符串，通常用于分布式或无状态的系统中。Token 一般由服务器生成，包含用户的身份信息、权限和有效期等信息。常见的 token 是 JWT（JSON Web Token）。

特点：

  • 存储位置：客户端，通常保存在 localStorage 或 sessionStorage 中。
  • 生命周期：通常由服务器控制，token 一旦过期，用户需要重新认证。Token 可以在多个系统间传递，不依赖于服务器端的会话数据。
  • 无状态：token 是自包含的（包括了认证信息），所以服务器不需要保存会话信息。每次请求时，token 会携带在请求中，服务器根据 token 来验证身份。

生活中的例子：

可以将 token 比作 银行取款凭证。你去银行取钱时，工作人员会给你一个凭证，凭证上写有你的取款金额和有效期。凭证本身就包含了所有的信息，你只要出示凭证，银行就知道你是谁，你能取多少钱。这个凭证不需要银行存储数据，因为凭证本身是自包含的。

常见使用：

  • OAuth 2.0 等认证协议中广泛使用 token 来验证用户身份。
  • API 请求中的身份验证，如使用 Bearer Token 进行 API 认证。
  • 移动应用中，通常使用 token 来管理用户会话。

生活中场景：一个在线购物网站

背景：

假设你正在一个在线购物网站上浏览商品并添加到购物车。为了确保你能顺利地购买商品，网站需要记住你的身份、购物车内容等。

1. Cookie（记住你的登录信息）

当你第一次访问网站时，服务器通过 Cookie 将一些信息（比如用户 ID）存储在你的浏览器中，这样下次你访问时，服务器就能知道你是谁。Cookie 是存储在浏览器中的小数据片段，通常包含用户的登录信息、偏好设置等。

  • 工作原理：你访问网站时，服务器检查你的 Cookie，发现你已经登录，并且返回相应的用户信息（比如购物车商品、用户名等）。
Cookie 的角色：
  • 保存你在不同页面间的身份信息。
  • 记录你的一些偏好设置（比如选择的商品、浏览记录等）。

2. Session（确保你的购物车和操作状态）

当你点击 “登录” 或者 “添加商品到购物车” 时，网站会在服务器端创建一个 Session。Session 用于记录和跟踪你的活动（例如你已登录、购物车中有多少商品等）。Session 会生成一个唯一的 ID（通常是随机的），并通过 Cookie 发送给浏览器。

  • 工作原理：每次你执行操作时，浏览器会把这个唯一的 Session ID 发送给服务器，服务器根据该 ID 找到你的会话数据（例如购物车内容、登录状态等）。
Session 的角色：
  • 保证你在网站上做的任何操作（如加入购物车）不会丢失。
  • 保证你退出时，服务器能清除相关的会话数据。

3. Token（API 调用时的认证）

当你在网站上进行一些需要身份验证的操作（例如支付、查看订单历史等），特别是在使用 RESTful API 或进行跨域请求时，使用 Token（如 JWT）进行认证和授权。

  • 工作原理：你登录时，服务器会生成一个 Token，该 Token 包含了你的身份信息（例如用户名、角色等），并返回给客户端。之后每次进行 API 请求时，客户端将 Token 放在请求头中，服务器通过验证 Token 来确定你是否有权限执行该操作。
Token 的角色：
  • 用于跨多个系统或服务的身份验证（例如支付系统、推荐系统等）。
  • 保证请求的有效性，防止 CSRF 和伪造请求。

为什么不仅仅依赖 Cookie，而需要 Session 和 Token？

1. Cookie 不足以解决所有问题

虽然 Cookie 可以用来存储用户的身份信息，但它有一些缺点：

  • 安全性问题：如果仅仅依赖 Cookie，用户的敏感信息（如会话数据）会暴露在浏览器中，容易被攻击者篡改（如盗用或窃取）。
  • 容易泄漏：因为 Cookie 会自动随每个请求一起发送到服务器，如果你在一个不安全的网络环境中，恶意攻击者可能会窃取你的 Cookie，从而伪造你的身份。
  • 过期管理：Cookie 需要有明确的过期时间，而 Session 是由服务器管理的，通常更灵活，能在一定时间内自动过期。

2. Session 解决了安全性和管理问题

  • 服务器端存储：Session 存储在服务器端，服务器通过一个唯一的 Session ID 来识别用户。在服务器端维护用户信息而不是存储在浏览器中，更加安全。
  • 动态控制：服务器可以随时控制 Session 的有效期，比如用户一定时间内没有操作，服务器会主动销毁 Session。
  • 跨页面存储：不同于 Cookie，Session 可以在整个会话期间跨页面和请求共享数据。

3. Token 提供了跨域和跨系统的认证能力

  • 跨域认证：在现代 Web 应用中，尤其是 SPA（单页面应用）和微服务架构下，前端和后端常常是分开的。这时，Token 尤其有用，因为它不需要依赖于服务器存储用户信息，且可以跨多个系统使用。
  • 无状态认证：相比于需要每次都查询服务器的 Session，Token 是自包含的，携带了用户的认证信息，因此每个请求都可以独立验证。

总结

特性 Session Cookie Token
存储位置 服务器端 客户端（浏览器） 客户端（如 localStorage）
生命周期 用户关闭浏览器或 session 过期 可以设置过期时间，默认浏览器关闭清除 由服务器设置有效期
大小限制 通常无大小限制 4KB 无明确限制，但通常较大
安全性 只在服务器可见，较为安全 客户端存储，容易被篡改 可能被伪造，需加密传输
应用场景 登录状态、临时信息存储 记录用户偏好、会话信息 用户认证、API认证
