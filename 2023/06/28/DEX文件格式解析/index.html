纸上得来终觉浅

DEX的由来

阶段 描述
Java与移动设备的限制 标准的Java字节码（JVM字节码）在移动设备上运行效率低，尤其在内存和处理器资源受限的环境中。
Dalvik虚拟机的诞生 为了解决性能问题，Android团队开发了Dalvik虚拟机（DVM），用于替代标准的Java虚拟机（JVM），专为移动设备优化。
DEX文件格式 DEX文件（Dalvik Executable）是Dalvik虚拟机的字节码格式，采用了紧凑的编码方式，多个类文件被合并成一个DEX文件，优化内存和存储效率。
优化特性 1. 紧凑的字节码格式，减少内存占用。 2. 多个类文件合并为一个DEX文件，减少I/O操作。 3. 内存管理优化，适应移动设备资源限制。
ART虚拟机的演变 从Android 5.0开始，Dalvik虚拟机被Android Runtime（ART）替代，ART优化了应用的执行速度，但仍支持DEX文件格式，直接运行或将其转换为更高效的运行时格式。

DEX 文件中的各部分结构与对应的示例：

1. 文件头部（Header）
  • magic：标识 DEX 文件版本，例如 0x64 65 78 0a 30 33 35 00 表示 “dex\n035\0”。
  • file_size：表示整个 DEX 文件的大小。
  • string_ids_size、string_ids_off：字符串标记的数量和偏移位置。
  • type_ids_size、type_ids_off：类型标记的数量和偏移位置。
  • method_ids_size、method_ids_off：方法标记的数量和偏移位置。
  • class_defs_size、class_defs_off：类定义的数量和偏移位置。
2. 字符串标记（String IDs）

String IDs 表存储了所有字符串，例如类名、字段名、方法名等。每个 String ID 表项指向字符串数据的偏移地址。对于 Person 类的例子：

  • "Person"：表示类名。
  • "name"：表示字段名称。
  • "age"：表示字段名称。
  • "getName"、"setName"：表示方法名称。
  • "getAge"、"setAge"：表示方法名称。
  • "Ljava/lang/String;"：表示 String 类型的描述符。
  • "I"：表示 int 类型的描述符。

这些字符串以 UTF-8 编码存储，并在 String IDs 中保存其偏移。

3. 类型标记（Type IDs）

Type IDs 表存储类、基本类型和数组类型。每个 Type ID 指向 String IDs 中的一个字符串，表示具体的类型。对于 Person 类：

  • LPerson;：表示 Person 类，指向 String IDs 中的 "Person"。
  • Ljava/lang/String;：表示 String 类型，指向 String IDs 中的 "Ljava/lang/String;"。
  • I：表示 int 类型，指向 String IDs 中的 "I"。
4. 方法原型标记（Proto IDs）

Proto IDs 表存储方法的原型信息，包括方法的返回类型和参数类型。例如，在 Person 类中：

  • public Person(String name, int age) 的方法原型：

    • 参数类型列表为 [Ljava/lang/String;, I（即一个 String 和一个 int）。
    • 返回类型为 V（表示 void，因为构造函数没有返回值）。

  • public String getName() 的方法原型：

    • 参数类型为空。
    • 返回类型为 Ljava/lang/String;（表示返回值是 String 类型）。

每个方法原型指向 Type IDs 表中对应的类型，并通过 Proto IDs 表项记录返回类型和参数类型。

5. 字段标记（Field IDs）

Field IDs 表存储类中的所有字段信息，包括字段的类型、字段名称和所属类。对于 Person 类中的字段：

  • name：

    • class_idx：指向 Type IDs 表中的 Person。
    • type_idx：指向 Type IDs 表中的 String 类型。
    • name_idx：指向 String IDs 表中的 "name"。

  • age：

    • class_idx：指向 Type IDs 表中的 Person。
    • type_idx：指向 Type IDs 表中的 int 类型。
    • name_idx：指向 String IDs 表中的 "age"。
6. 方法标记（Method IDs）

Method IDs 表存储类中的所有方法信息，包括方法名称、方法原型和所属类。对于 Person 类中的方法：

  • getName：

    • class_idx：指向 Type IDs 表中的 Person。
    • proto_idx：指向 Proto IDs 表中的原型（返回 String，无参数）。
    • name_idx：指向 String IDs 表中的 "getName"。

  • setName：

    • class_idx：指向 Type IDs 表中的 Person。
    • proto_idx：指向 Proto IDs 表中的原型（参数为 String，返回 void）。
    • name_idx：指向 String IDs 表中的 "setName"。
7. 类定义（Class Definitions）

Class Definitions 表存储类的结构信息，包括类的访问权限、父类、接口、类中的字段和方法等信息。对于 Person 类：

  • class_idx：指向 Type IDs 表中的 Person 类。
  • access_flags：表示类的访问权限，例如 public、final 等。
  • superclass_idx：指向父类（通常是 Ljava/lang/Object;，即 Object 类）。
  • interfaces_off：指向实现的接口（如果有）。
  • class_data_off：指向类的数据（包括类的字段和方法）。
8. 数据区（Data Section）

数据区包含方法的字节码实现、静态字段的初始值、注解等。对于 Person 类的构造方法和 getName 方法，数据区存储了它们的具体字节码：

  • 字节码（Code Items）：方法的字节码，包含指令、寄存器数、异常处理表等。例如，getName() 方法的字节码实现会加载 name 字段并返回其值。
  • 字符串数据：存储了所有字符串内容，如 "Person"、"name"、"getName" 等。
  • 类型列表：存储方法的参数类型列表。

要注意解析过程中的Uleb128结构和mutf-8结构

可变长度的类型其实都是基于 LEB128(Little-Endian Base) 类型的，可以用于表示 32 位大小的 int 数字，其根据所要表示的数字的大小来选择合适的长度。如下图所示，其中每个字节的最高位表示是否使用下一个字节，1 表示使用，0 表示不使用。故而每个字节其实只有 7 个有效的 bit 位用来表示相应的数字。如果有一个 LEB128 类型的变量使用了 5 个字节，并且第五个字节的最高位为 1 ，那说明出现了问题。

map_list结构

Dalvik 可执行文件格式 | Android Open Source Project

Android .dex file format

在文件头解析中简单提到了mapOff这个偏移量，当时没太在意，之后在解析Frida-dexdump脱壳脚本时，才发现遗漏了这个比较重要的结构，故做补充。

mapOff是DexMapList结构的文件偏移。

map_off（映射列表偏移）是DEX文件头（Header）中的一个关键字段，用于指向 map_list段 的起始偏移地址。此段记录了DEX文件中所有数据结构的 类型、数量及内存偏移，是DEX格式的全局索引表。

map_list定义格式

map_offset是map_list的文件偏移，通常在dex文件末尾 map_list有两个成员,map_item有四个成员

DexMapList

DexMapItem

常见类型标识

正如官方文档所言，map_off处于data section中。map_list记录了整个dex文件拥有哪些Item Type，以及它们各自在文件中的偏移地址。放置这个map_list的目的是 可以更快速的迭代整个dex文件。比方说，我们需要获得String：

  • 需要先找到string_ids_off(第一步)，再根据size把所有的string_ids（其实就是String的偏移地址）拿到（第二步），然后根据ids去data section中取String（第三步）；
  • 如果有map_list，找到map_off（第一步），就可以定位到string_data_item（maplist中的类型值是0x2002），得到String的个数和偏移地址（第二步），从偏移地址取String（第三步）。

好像都需要三步，没有什么变化呀！但是，如果是迭代dex文件，后续获取其它类型数据时，不再需要取map_off了（省了一步），由此可见，map_list的作用相当于减少了1/3的处理时间，这使得Android系统在加载dex文件时获得优异的性能提升。
