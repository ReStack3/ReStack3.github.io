Java的反射机制

Java反射机制是指在程序运行时动态加载类并获取类的详细信息，从而可以操作类或对象的属性和方法。反射机制的核心是通过 Class 对象来实现的。当Java虚拟机（JVM）获得类的 Class 对象后，可以通过这个对象获取类的构造函数、方法、字段等信息，并且可以在运行时创建对象、调用方法或访问属性。这使得Java程序在运行时具有更高的灵活性和动态性。

可以把class文件想象成类的结构化目录，该类的成员变量和函数等都被分门别类的存在class的不同局域位置。而多个不同的class又组成了一个class目录。并于通过查询该目录找到对应的类的所有信息

常规实例化

想象你是一个图书馆管理员，编译时你知道所有书籍的名称和作者（例如，”Java 编程”）。当有人请求这本书时，你直接去书架上拿书。

Java是一种先编译后运行的语言，这意味着在编译时，程序中所有对象的类型和结构都是固定的。然而，在某些情况下，程序可能需要在运行时动态地加载某些类。比如，在插件化架构中，应用程序可以根据用户的选择或需求动态加载不同的模块，而这些模块在编译时并不存在。

反射机制允许开发者在运行时获取类的 Class 对象后【在 Java 中，类的 Class 对象是反映 Java 类的内部结构和行为的特殊对象。每个 Java 类、接口和枚举类型在运行时都会由 JVM 创建一个对应的 Class 对象，这个 Class 对象包含了关于该类的所有信息，例如类的名称、父类、实现的接口、方法、字段、构造函数等。】，可以利用这个对象来获取该类的构造函数、方法和字段等信息。基于这些信息，可以创建该类的实例、调用其方法或访问其属性，而不需要在编译时硬编码类的名称。

反射实例化

现在，假设你有一个应用程序，它可以根据用户的请求动态获取书籍。用户可能输入书名，而你不知道这个书名是否在书架上。你通过输入的书名查找图书馆数据库（使用反射），并在运行时决定是否能够找到这本书。如果书存在，你可以将它拿出来；如果书不存在，你将会处理一个错误。

反射的优缺点

1、优点：

（1）在运行时获得类的各种内容，进行反编译，对于Java这种先编译再运行的语言，能够让我们很方便的创建灵活的代码，这些代码可以在运行时装配，无需在组件之间进行源代码的链接，更加容易实现面向对象。

2、缺点：

（1）反射会消耗一定的系统资源，因此，如果不需要动态地创建一个对象，那么就不需要用反射；

（2）反射调用方法时可以忽略权限检查，因此可能会破坏封装性而导致安全问题。

主要 ClassLoader 类型

BootClassLoader 由 Java 实现的 ClassLoader 内部类，用于加载 Android Framework 层的类文件。 Android Framework 层的 class 文件 单例类，仅能在同一包中访问，应用程序无法直接调用。

Custom ClassLoader 用户自定义的类加载器，可以根据需求实现特定的类加载逻辑。 用户自定义的类加载逻辑 根据实现情况，通常应用程序可以直接使用。

双亲委派模型

在初识JVM - The Peak Tower中有对java的双亲委派模型做了一定的了解。

本质上 Android 和传统的 JVM 是一样的，也需要 ClassLoader 将目标类加载到内存，类加载器之间也复合双亲委派模型。但是在 Android 中，ClassLoader的加载细节有略微的差别。在 Android 虚拟机里是无法直接运行 .class 文件，Android 中会将所有的 .class 文件转化为一个 .dex 文件。Android 将加载 .dex 文件的实现封装在 BaseDexClassLoader 中，一般我们使用它的两个子类：PathClassLoader 和 DexClassLoader。

dex加载运行整体步骤

加载 classes.dex 文件

当应用程序启动时，系统会通过 ClassLoader 读取 APK 中的 classes.dex 文件。这个文件包含了应用的字节码。

初始化 DexClassLoader 或 PathClassLoader

Android 提供了不同类型的 ClassLoader（如 DexClassLoader 和 PathClassLoader）用于加载 .dex 文件。这些加载器在初始化时会指向要加载的 classes.dex 文件的路径。

解析 .dex 文件格式

.dex 文件的格式与 Java 的 .class 文件不同。它的结构包括：

  • 魔数（Magic Number）：用于标识文件格式。
  • 版本号：表示 .dex 文件的版本。
  • 字符串表：存储字符串常量的索引。
  • 类型表：存储类和接口的描述信息。
  • 方法表：描述所有方法的信息。
  • 字段表：描述所有字段的信息。

读取数据

ClassLoader 使用特定的解析器（如 Dex 类）读取 .dex 文件中的数据结构。通过索引和表的结构，ClassLoader 能够快速定位到每个类、字段和方法的描述。

查找和加载类

当需要加载一个类时，ClassLoader 会：

  • 查找字符串表，根据给定的类名找到类的索引。
  • 解析类型表，获取类的具体描述，包括父类、实现的接口等信息。
  • 解析方法表和字段表，获取类的方法和字段的详细信息。

检查已加载类

在加载之前，ClassLoader 会先检查其缓存，以查看类是否已经加载（通过 findLoadedClass() 方法）。如果已加载，则直接返回该类；如果未加载，继续执行后续加载操作。

生成类对象

一旦找到类的信息，ClassLoader 会调用 defineClass() 方法将字节码转换为 Java 对象。此时，JVM 会为该类分配内存，并初始化它的静态变量。

字节码验证

在加载过程中，系统会进行字节码验证，以确保类的安全性和有效性，防止恶意代码的执行。

JIT/AOT 编译

在 Dalvik 中，首次调用方法时会进行即时编译（JIT），将字节码编译为机器码。在 ART 中，应用安装时会将 .dex 文件的字节码编译为机器码（AOT），提高运行效率。

一/二代壳【整体加固】实现的关键点

接下来，简单介绍在脱壳程序中的几个关键步骤的意义和作用。

  1. 创建源程序的DexClassLoader实例替换LoadedApk中的mClassLoader。

源程序要依靠ClassLoader动态加载的，而ClassLoader存放了应用所需的类库当需要加载某些类时，就会开始双亲委托模式寻找，而父类加载器以及祖先加载器里面肯定是没有源程序所需要的类的，因而也不可能加载成功，所以我们就需要创建一个新的DexClassLoader来替换壳程序的ClassLoader，由于里面包含源程序的类，这样一来，就可以成功加载源程序。

  1. 创建源程序的Application实例替换壳程序的Application实例。

修正加载器

简单分析

如何加载dex文件？

Android里边的BaseDexClassLoader可以实现在运行的时候加载在编译时未知的dex文件，经过此加载器的加载，ART虚拟机内存中会形成相应的数据结构，对应的dex文件也会由mmap映射到虚拟内存当中，通过此加载器的loadClass(String className, boolean resolve)方法就可以得到类的Class对象，从而可以使用该类。

查看源码可以看到PathClassLoader是继承自BaseDexClassLoader的，而PathClassLoader还有另外两个兄弟: InMemoryDexClassLoader以及DexClassLoader，而壳程序很多都使用了这两个类加载器来加载解密后的dex文件。其中InMemoryDexClassLoader是Android8.0以后新增的类，可以实现所谓的”不落地加载”。

如何让修改加载之后的Application进入后续的加载流程？

后续的加载流程指的就是app组件（比如Activity）的加载，而加载组件时，使用的是加载应用程序的ClassLoader。

如若不做任何处理，仅仅在attachBaseContext方法中使用上面讲的某个类加载器对dex加载，后续加载源程序的组件时会出现ClassNotFoundException的错误，为什么会这样？

这是因为如果仅仅在attachBaseContext方法中使用类加载器加载dex，之后加载组件时使用的ClassLoader和我们使用的加载器不同，并且，加载组件的ClassLoader通过双亲委派模型发现没有人能加载组件类（因为组件类在我们的dex中），导致ClassNotFoundException。

还记得BaseDexClassLoader吗，其有一个DexPathList，记录了已加载的dex文件路径。

加载组件时对应的BaseDexClassLoader的DexPathList是没有源程序的dex路径的，如果尝试让BaseDexClassLoader加载不在这个列表中的类，就会报ClassNotFoundException。

因此有两种方法可以解决这个问题。

  1. 既然使用的加载器不同，那么改成相同的不就行了。

    通过反射获取到LoadedApk，修改其mClassLoader为我们加载dex文件的ClassLoader实例，这样后续试图加载组件类的时候，就能找到相应的类。

  2. 通过打破原有双亲委派关系，添加我们的ClassLoader进入关系网。

    原先的mClassLoader是PathClassLoader，其在双亲委派关系中的父亲是BootClassLoader，所以只要将我们的ClassLoader添加进他们两个之间即可。也就是将PathClassLoader的父亲设置为我们自己的ClassLoader，再将我们自己的ClassLoader的父亲设置为BootClassLoader

修正并加载源Application

简单分析

为什么要替换Application？

因为Application做为整个应用的上下文，它的生命周期就是整个应用程序的生命周期，它还会与Activity、Service等组件有交互关系。对于壳程序和源程序，它们对应的Application实例不一定相同，为了维护应用的正常生命周期，所以需要进行替换。可以认为Application就代表着整个APP，Application都不同，那运行的App能是一样的吗？

为什么选择Application类作为脱壳代码执行的地方？

因为Application是整个应用程序的入口点，它的创建早于任何一个Activity、Service的创建（ContentProvider例外），因此壳程序必须在Android系统启动组件之前运行，完成源程序的解密和动态加载，否则会使程序出现加载类失败的异常，所以选择这里作为脱壳代码执行的地方。

其次，Application类的attachBaseContext()方法比onCreate()方法先执行。

全在attachBaseContext()方法或者全在onCreate()方法中执行都行的！但在attachBaseContext()方法执行会有缺陷，就是ActivityThread实例中的mInitialApplication又会被壳程序的Application替换掉。

怎么替换Application？

需要将壳程序的Application（也就是mApplication）从ActivityThread实例中的mAllApplications中删除。之后设置LoadedApk实例中的mApplication字段为null，这样才能调用LoadedApk类的makeApplication()方法创建源程序的Application实例（也是反射，需要获取到源Application类名，可以通过在壳程序的ActivityManifest.xml中使用<meta-data>标签填写并在代码中获取）。当然ContentProvider也会有Application实例，也是需要替换的。最后运行源程序的Application实例的onCreate()方法，这样源程序才开始它的运行生命周期。

如果源程序也有自定义的Application，我们就需要重新makeApplication，进入到源程序的Application，保证程序的完整生命周期。
