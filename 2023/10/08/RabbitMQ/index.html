RabbitMQ基本概念

1. 什么是 RoutingKey（路由键）？

RoutingKey 是 RabbitMQ 中用于将消息从 交换机（Exchange） 路由到 队列（Queue） 的关键字。它的作用类似于邮件的“收件人地址”，决定了消息最终会被发送到哪个队列

  • 在 Direct 和 Topic 类型的交换机中，RoutingKey 用于匹配绑定规则（Binding Key），从而决定消息的路由目标。
  • 在 Fanout 类型的交换机中，RoutingKey 被忽略，消息会被广播到所有绑定的队列。
  • 在 Headers 类型的交换机中，RoutingKey 被忽略，消息的路由由消息头（Headers）决定。

2. Binding（绑定）？

Binding 是 RabbitMQ 中 交换机（Exchange） 和 队列（Queue） 之间的关联规则。它定义了交换机如何将消息路由到队列。决定消息从交换机到队列的路由规则。

  • 交换机（Exchange）：消息的入口。
  • 队列（Queue）：消息的存储位置。
  • 绑定键（Binding Key）：用于匹配 RoutingKey 的规则（在 Direct 和 Topic 类型的交换机中有效）。

3. 交换器4种类型？

RabbitMQ 支持以下 4 种类型的交换机：

Direct（直连交换机）：

根据 RoutingKey 精确匹配 Binding Key，将消息路由到对应的队列。适用场景：点对点消息传递。

Fanout（广播交换机）：

忽略 RoutingKey，将消息广播到所有绑定的队列。适用场景：广播消息。

Topic（主题交换机）：

根据 RoutingKey 和 Binding Key 的模式匹配（支持通配符 * 和 #），将消息路由到匹配的队列。适用场景：复杂的消息路由。

Headers（头交换机）：

忽略 RoutingKey，根据消息头（Headers）的属性匹配 Binding Key，将消息路由到匹配的队列。适用场景：基于消息属性的路由。

4. 生产者消息运转？

生产者消息的运转过程如下：

  1. 创建连接：生产者连接到 RabbitMQ 服务器。
  2. 创建通道：在连接上创建通道（Channel）。
  3. 声明交换机：声明一个交换机（可选，如果交换机已存在则不需要）。
  4. 发送消息：将消息发送到交换机，并指定 RoutingKey。
  5. 路由消息：交换机根据 RoutingKey 和绑定规则将消息路由到队列。
  6. 持久化：如果消息和队列是持久化的，消息会被写入磁盘。
  7. 确认机制：如果启用了生产者确认机制，RabbitMQ 会返回确认（ACK）或否定确认（NACK）。

5. 消费者接收消息过程？

消费者接收消息的过程如下：

  1. 创建连接：消费者连接到 RabbitMQ 服务器。
  2. 创建通道：在连接上创建通道（Channel）。
  3. 声明队列：声明一个队列（可选，如果队列已存在则不需要）。
  4. 绑定队列：将队列绑定到交换机（可选，如果绑定已存在则不需要）。
  5. 消费消息：从队列中接收消息。
  6. 处理消息：消费者处理消息。
  7. 确认消息：消费者手动发送确认（ACK）给 RabbitMQ，表示消息已成功处理。
  8. 重试机制：如果消息处理失败，消费者可以发送否定确认（NACK）或将消息重新放回队列。

6. 队列结构

RabbitMQ 的队列是一个先进先出（FIFO）的数据结构，用于存储消息。队列的主要属性包括：

  • 名称：队列的唯一标识。
  • 持久化：队列是否持久化（重启后是否仍然存在）。
  • 独占：队列是否只能被一个连接使用。
  • 自动删除：当没有消费者时，队列是否自动删除。
  • 参数：队列的其他配置参数（例如消息的 TTL、死信队列等）。

7. 死信队列？

死信队列（Dead Letter Queue, DLQ） 是用于存储无法被正常消费的消息的队列。

导致消息进入死信队列的几种原因包括：

  1. 消息被拒绝：消费者发送 NACK 或 Reject，且未设置重新入队。
  2. 消息过期：消息在队列中的存活时间（TTL）到期。
  3. 队列满：队列达到最大长度，无法再接收新消息。
  4. 队列被删除：消息所在的队列被删除。

8. 延迟队列？

延迟队列（Delayed Queue） 是用于实现延迟消息的队列。消息在延迟时间到达后才会被投递到队列。使用 RabbitMQ 的插件（如 rabbitmq_delayed_message_exchange）。设置消息的延迟时间。适用场景：定时任务、订单超时等。

9. 优先级队列？

优先级队列（Priority Queue） 是支持消息优先级的队列。高优先级的消息会被优先消费。

声明队列时设置最大优先级。发送消息时指定优先级。

适用场景：

  • 订单在十分钟之内未支付则自动取消
  • 新创建的店铺，如果在十天内都没有上传过商品，则自动发送消息提醒
  • 用户注册成功后，如果三天内没有登陆则进行短信提醒
  • 用户发起退款，如果三天内没有得到处理则通知相关运营人员
  • 预定会议后，需要在预定的时间点前十分钟通知各个与会人员参加会议

10. 事务机制？

事务机制 是 RabbitMQ 提供的一种确保一组操作原子性的机制。

  • 开启事务。
  • 发送一组消息。
  • 提交事务或回滚事务。

缺点：性能较差，通常不推荐使用。

RabbitMQ的工作模式

简单模式

简单模式是最基本的消息队列模式，包含以下角色：

  • 生产者（Producer）：发送消息到队列。
  • 队列（Queue）：存储消息的缓冲区。
  • 消费者（Consumer）：从队列中接收并处理消息。

一个生产者对应一个队列，一个队列对应一个消费者。消息是单向传递的，生产者发送消息到队列，消费者从队列中接收消息。

适用场景：简单的任务分发场景，例如日志收集、任务队列等。

消息的消费者(consumer) 监听消息队列,如果队列中有消息,就消费掉,消息被拿走后,自动从队列中删除(隐患 消息可能没有被消费者正确处理,已经从队列中消失了,造成消息的丢失，这里可以设置成手动的ack,但如果设置成手动ack，处理完后要及时发送ack消息给队列，否则会造成内存溢出)。

工作队列模式

工作队列模式是简单模式的扩展，允许多个消费者从同一个队列中接收消息。

  • 一个生产者对应一个队列，多个消费者共享同一个队列。
  • 消息会被均匀地分发给消费者（轮询分发）。
  • 支持消息确认机制（ACK），确保消息被成功处理。

适用场景：任务分发场景，例如多个工作进程处理任务队列中的任务。(隐患：高并发情况下,默认会产生某一个消息被多个消费者共同使用,可以设置一个开关(syncronize) 保证一条消息只能被一个消费者使用)。

发布/订阅模式

发布/订阅模式通过 交换机（Exchange） 将消息广播给多个队列。

  • 生产者将消息发送到交换机，交换机将消息广播到所有绑定的队列。
  • 每个队列可以有一个或多个消费者。
  • 交换机类型为 Fanout，忽略路由键（Routing Key）。

适用场景：广播消息场景，例如新闻推送、系统通知等。

路由模式

路由模式通过 交换机（Exchange） 和 路由键（Routing Key） 将消息发送到指定的队列。

  • 生产者将消息发送到交换机，并指定路由键。
  • 交换机根据路由键将消息路由到匹配的队列。
  • 交换机类型为 Direct，精确匹配路由键。

适用场景：根据条件分发消息的场景，例如日志分类（ERROR、INFO、WARN）。

主题模式

主题模式是路由模式的扩展，支持更灵活的路由规则。

  • 生产者将消息发送到交换机，并指定路由键。
  • 交换机根据路由键的模式匹配将消息路由到匹配的队列。
  • 交换机类型为 Topic，支持通配符匹配路由键（* 匹配一个单词，# 匹配零个或多个单词）。

适用场景：复杂的消息分发场景，例如根据消息类型或来源进行分发。

RabbitMQ消息的可靠传输

生产者到RabbitMQ

生产者确认机制

注意：事务机制和 生产者确认机制 机制是互斥的，两者不能共存，会导致 RabbitMQ 报错。

  • 生产者将消息发送到 RabbitMQ 后，等待服务器的确认（ACK）。
  • 如果服务器成功接收到消息，会返回一个确认消息。
  • 如果服务器未能接收到消息（例如队列已满或服务器崩溃），会返回一个否定确认（NACK）。

事务机制

事务机制用于确保一组操作的原子性。

  • 生产者开启事务，发送一组消息。
  • 如果所有消息都成功发送，提交事务。
  • 如果任何消息发送失败，回滚事务。

RabbitMQ自身

持久化机制

持久化配置可以和生产者确认机制配合使用，可以在消息持久化磁盘后，再给生产者发送一个Ack信号。

这样，如果消息持久化磁盘之前，rabbitMQ阵亡了，那么生产者收不到Ack信号，生产者会自动重发。

持久化机制用于确保消息在 RabbitMQ 服务器重启后不会丢失。

  • 持久化队列：将队列声明为持久化（durable=true），确保队列在服务器重启后仍然存在。
  • 持久化消息：将消息的 deliveryMode 设置为 2，确保消息被写入磁盘。

消息持久化，当然前提是队列必须持久化

  • 声明队列必须设置持久化 durable 设置为 true.
  • 消息推送投递模式必须设置持久化，deliveryMode 设置为 2（持久）。
  • 消息已经到达持久化交换器。
  • 消息已经到达持久化队列。

RabbitMQ到消费者

消费者确认机制

消费者确认机制用于确保消息被成功处理。

  • 消费者从队列中接收消息后，手动发送确认（ACK）给 RabbitMQ 服务器。
  • 如果消费者未能处理消息（例如发生异常），可以发送否定确认（NACK）或将消息重新放回队列。

死信队列机制

死信队列机制用于处理无法被正常消费的消息。

  • 将无法被消费的消息发送到死信交换机（DLX）。
  • 死信交换机将消息路由到死信队列（DLQ），进行进一步处理。

常见问题

1. 如何解决消息队列的延时以及过期失效问题？

延时问题

使用延迟队列：

通过 RabbitMQ 的插件（如 rabbitmq_delayed_message_exchange）实现延迟队列。发送消息时设置延迟时间，消息在延迟时间到达后才会被投递到队列。

适用场景：定时任务、订单超时等。

过期失效问题

设置消息的 TTL（Time-To-Live）：

在发送消息时设置消息的过期时间。如果消息在队列中超过 TTL，会被自动丢弃或发送到死信队列。

消息队列满了以后该怎么处理？

设置队列的最大长度：

在声明队列时设置 x-max-length 参数，限制队列的最大消息数量。当队列满时，新消息会被丢弃或发送到死信队列。

使用死信队列：当队列满时，将消息发送到死信队列进行处理。

增加消费者：增加消费者以提高消息的处理速度，避免队列积压。

3. 有几百万消息持续积压几小时，说说怎么解决？

临时扩容：增加消费者数量，提高消息的处理速度。增加 RabbitMQ 集群节点，提高消息的处理能力。

批量处理：修改消费者逻辑，支持批量处理消息。

消息丢弃：如果消息允许丢失，可以丢弃部分旧消息，优先处理新消息。

数据分流：将消息分流到多个队列中，分散压力。

4. 设计 MQ 思路？

设计一个消息队列系统时，需要考虑以下方面：

  1. 消息模型：
    • 支持点对点、发布/订阅等消息模型。
  2. 高可用性：
    • 使用集群和镜像队列，确保系统的高可用性。
  3. 消息可靠性：
    • 支持消息持久化、生产者确认、消费者确认等机制。
  4. 性能优化：
    • 支持批量处理、异步确认等机制，提高性能。
  5. 扩展性：
    • 支持水平扩展，动态增加节点。
  6. 监控和告警：
    • 提供监控和告警功能，及时发现和处理问题。

5. 如何保证 RabbitMQ 消息的顺序性？

单队列单消费者：将需要保证顺序的消息发送到同一个队列，并由单个消费者处理。

消息分组：使用 RoutingKey 或消息头将消息分组，确保同一组消息由同一个消费者处理。

全局序列号：在消息中添加全局序列号，消费者根据序列号处理消息。

6. RabbitMQ 消息丢失的情况有哪些？

生产者丢失：消息未成功发送到 RabbitMQ。

解决方案：使用生产者确认机制。

RabbitMQ 丢失：消息未持久化，服务器重启后丢失。

解决方案：使用持久化机制。

消费者丢失：消费者未成功处理消息。

解决方案：使用消费者确认机制。

7. RabbitMQ 中消息可能有的几种状态？

  • 未确认：消息已被消费者接收，但未确认。
  • 已确认：消息已被消费者成功处理并确认。
  • 已拒绝：消息被消费者拒绝。
  • 已过期：消息在队列中超过 TTL。
  • 死信：消息被发送到死信队列。

8. 为什么不应该对所有的 message 都使用持久化机制？

性能开销：持久化消息需要写入磁盘，性能较差。

存储成本：持久化消息占用更多的存储空间。

适用场景：只有对可靠性要求高的消息才需要持久化。

9. 如何保证消息不被重复消费？或者说，如何保证消息消费时的幂等性？

幂等性设计：消费者逻辑设计为幂等操作，即使重复消费也不会产生副作用。

唯一标识：在消息中添加唯一标识（如 UUID），消费者记录已处理的消息 ID。

数据库约束：使用数据库的唯一约束，防止重复处理。

10. 消息基于什么传输？

  • 协议：
    • RabbitMQ 基于 AMQP（Advanced Message Queuing Protocol）协议传输消息。
  • 传输方式：
    • 消息通过 TCP 连接传输，支持持久化和非持久化消息。

11. 消费者某些原因无法处理当前接受的消息如何来拒绝？

当消费者无法处理当前接收的消息时，可以通过以下方式拒绝消息：

2.1 否定确认（NACK）

作用：告诉 RabbitMQ 消息未被成功处理。

参数：

  • multiple：是否拒绝多条消息。
  • requeue：是否将消息重新放回队列。

2.2 拒绝消息（Reject）

作用：告诉 RabbitMQ 消息未被成功处理。

参数：

  • requeue：是否将消息重新放回队列。

2.3 死信队列

作用：将无法处理的消息发送到死信队列，进行进一步处理。

配置：在声明队列时设置死信交换机和死信队列。
