PE文件格式解析

PE文件格式概述

PE（Portable Executable）文件格式是Windows操作系统下可执行文件、动态链接库（DLL）和设备驱动程序的标准文件格式，由Microsoft设计并广泛应用于32位（PE32）和64位（PE32+）系统中。

其结构基于COFF（Common Object File Format）规范，通过分层组织将程序代码、数据、资源和元数据整合为单一文件。PE文件以DOS头（IMAGE_DOS_HEADER）开头，保留对旧版DOS系统的兼容性，随后通过e_lfanew字段定位到真正的PE头（IMAGE_NT_HEADERS），其中包含签名、文件头（IMAGE_FILE_HEADER）和可选头（IMAGE_OPTIONAL_HEADER）。

文件头定义机器类型、节表数量等基础属性，而可选头则详细说明入口点、内存对齐、镜像基址、子系统类型及数据目录表（DataDirectory）等关键信息。数据目录表指向16种核心数据结构（如导入表、导出表、资源表、重定位表等），这些结构支撑动态链接、资源管理和地址修正等功能。紧随头部的是节表（IMAGE_SECTION_HEADER），描述各节（如.text、.data、.rsrc等）的名称、内存偏移、文件大小及访问权限（读/写/执行），最终通过节的实际数据完成代码与数据的物理布局。

PE格式通过内存映射（Memory-Mapping）机制实现高效加载，结合重定位表和导入/导出表支持模块化编程，同时利用资源表管理非代码资产，是Windows生态中程序执行与扩展的基石。

典型PE文件布局（32位示例）

结构 偏移 大小（字节）

PE文件的两种状态

PE文件在生命周期中存在两种状态，分别对应不同的物理形态和用途：

磁盘存储状态（File Image）

  • 以完整的文件形式保存在硬盘中，按照PE文件格式规范严格排列
  • 各节区（Section）按文件对齐（File Alignment，通常512字节或4KB）存储
  • 包含所有头部信息、未解析的导入表和调试符号等辅助数据
  • 重定位表、资源数据等保持原始RVA（相对虚拟地址）形式
  • 特征：存在文件头冗余数据（如DOS Stub）、部分节区可能被压缩或加密

内存运行状态（Memory Image）

  • 由系统加载器映射到进程虚拟地址空间后的形态
  • 节区按内存对齐（Section Alignment，通常4KB）重新排布
  • 导入表被解析为实际函数地址（IAT填充），重定位表完成地址修正
  • 资源数据可通过内存API直接访问，调试信息可能被剥离
  • 特征：代码段获得执行权限，.bss节区被扩展为可写内存，部分节区（如.reloc）可能被丢弃

这两种状态的转换由Windows加载器（Loader）完成，涉及文件映射、地址修正、依赖解析等关键步骤，是PE文件执行的基础机制。

RVA与FOA的转换

在前面我们知道PE文件有两种状态:

  1. 磁盘状态：PE文件以紧凑的格式存储在磁盘上，各节(section)按文件对齐方式排列
  2. 内存状态：PE文件被加载到内存后，各节按内存对齐方式排列，且可能有不同的布局

由于这两种状态下PE文件的结构不同，导致同一数据在磁盘文件和内存中的位置偏移量不同,那么就存在不同的地址表示形式：

  1. RVA (Relative Virtual Address，相对虚拟地址)：相对于PE文件加载到内存后的基地址的偏移量，【FileAlignment：磁盘上节的对齐粒度(通常512字节或4KB)】
  2. FOA (File Offset Address，文件偏移地址)：PE文件在磁盘上的原始偏移地址。【SectionAlignment：内存中节的对齐粒度(通常4KB或64KB)】

从磁盘状态转化成内存状态就是所谓的RVA与FOA的转换:

RVA=内存地址-ImageBase

  1. SectionAlignment和FileAlignment是否相同，RVA是否在PE头中。===>如果相同，则 RVA=FOA
  2. 判断RVA在哪个节中：[节.VirtualAddress]<=ROV<=[节.VirtualAddress+当前节内存对齐后的大小(SectionAlignment)]
  3. FOA=节.PointerToRawData+差值(RVA-节.VirtualAddress)

PE文件整体结构解析

之前我们已经按照PE文件的整体结构对实际的PE文件进行了大致上的了解了，现在我们需要来看看每个结构的意义和作用。

DOS头

DOS头是PE文件结构中最开始的部分，全称为”MS-DOS头部”或”IMAGE_DOS_HEADER”。它是一个固定大小的数据结构，位于PE文件的最开头（偏移0处），主要作用是保持与早期MS-DOS系统的兼容性。DOS 程序包含了一个 DOS 头部和一个 DOS 程序体（ DOS 存根或 DOS 残留）。DOS 头是用来装载 DOS 存根用的。

MZ 头部

DOS MZ文件头就是一个结构体IMAGE_DOS_HEADER，其定义如下所示：

注意这里的关键字段：

  • **e_magic (2字节)**：DOS签名，固定为”MZ”（十六进制0x5A4D），用于标识这是一个有效的DOS可执行文件
  • **e_lfanew (4字节)**：最重要的字段，指示PE头（NT头）在文件中的偏移位置，现代PE文件中，这个值通常为0x00000080（128字节）
  • 其他DOS相关字段：如e_ss、e_sp、e_ip、e_cs等，用于16位DOS程序的执行，在现代Windows PE文件中这些字段基本不再使用

DOS 存根

在DOS头和PE头之间通常包含一个DOS存根程序（DOS Stub）：

  • 一段小型DOS可执行代码
  • 当文件在DOS下运行时显示”This program cannot be run in DOS mode”之类的提示
  • 大小不固定，由链接器决定
  • 可以通过修改e_lfanew的值来跳过

这里面的内容可以根据自己的需要随意的修改和添加，并不会影响文件的正常运行。

PE头是PE文件结构的核心部分，位于DOS头（通过e_lfanew字段定位）之后，它完整定义了PE文件的执行特征和内存布局。PE头由三个关键部分组成，形成一个层次化的结构体系：

注意：PE签名（Signature）

  • 4字节的魔数标识，固定为”PE\0\0”（十六进制0x00004550）
  • 作为PE文件的第二重验证标志（第一重是DOS头的”MZ”签名）
  • 若此签名损坏或不存在，系统会拒绝加载该可执行文件

关键字段解析：

  • 32位版本（224字节）：Magic值为0x010B
  • 64位版本（240字节）：Magic值为0x020B

  • 入口点：AddressOfEntryPoint（4字节）指定程序执行的起始RVA
  • 内存对齐：SectionAlignment（4字节）和FileAlignment（4字节）定义内存/磁盘对齐规则
  • 子系统：Subsystem（2字节）区分GUI/CUI/驱动等类型
  • 数据目录：DataDirectory[16]（每个8字节）指向导入表、导出表、重定位表等关键数据结构
  • 成员CheckSum表示校验和，是用来判断文件是否被修改的，它的计算方法就是文件的两个字节与两个字节相加，最终的值（不考虑溢出情况）就是校验和。

同时这里面在内存布局有着关键作用，PE头通过多个字段协同定义内存映射规则：

  • ImageBase：建议加载基地址（4/8字节）
  • SizeOfImage：整个PE映像占用的虚拟内存大小
  • SizeOfHeaders：所有头结构（DOS头+PE头+节表）的总大小
  • 数据目录：16个预定义的数据结构位置（如导入表RVA和大小）

DataDirectory 之前的结构体部分固定占 96 字节。

DataDirectory 数组

DataDirectory（数据目录）是PE文件结构中最重要的指针系统之一，它位于PE头的OptionalHeader末尾，作为PE文件与操作系统加载器之间的”通信枢纽”，指示了PE文件中关键数据结构的位置和大小。

导出表

导出表是PE文件中定义模块（通常是DLL）向外部暴露函数和数据的核心数据结构，它使其他程序能够通过名称或序号调用该模块提供的功能。作为PE文件数据目录(DataDirectory)的第一个条目（索引0），导出表在Windows模块动态链接中扮演着关键角色。

导出目录结构

导出表基础结构就是导出目录结构（IMAGE_EXPORT_DIRECTORY）

三表联动机制

导出表通过三个关键数组实现名称-地址映射：

  • 函数地址表（EAT）：存储所有导出函数的RVA
  • 函数名称表（ENT）：存储有名称导出的函数名称字符串RVA
  • 序号表（EOT）：存储名称对应在EAT中的索引

导出表查找流程

按名称查找：

  1. 在ENT中二分查找目标名称字符串
  2. 获取对应EOT中的索引值
  3. 通过索引从EAT获取函数RVA

按序号查找：

  1. 计算EAT索引 = 请求序号 - Base
  2. 直接访问EAT获取函数RVA
  3. 若索引超出范围或RVA为0则失败

导入表

导出表是PE文件（特别是DLL）中定义模块对外接口的核心数据结构，它建立了函数名称/序号与内存地址之间的映射关系。作为Windows动态链接机制的基础，导出表直接支持GetProcAddress()等关键API的功能实现。

导出目录头

导出表核心结构就是 导出目录头（IMAGE_EXPORT_DIRECTORY）

双表机制

每个导入DLL对应两个关键数组：

  • INT（Import Name Table）：存储函数名称/序号信息（只读）
  • IAT（Import Address Table）：运行时被加载器填充为实际地址（可写）

第一个成员指向的就是一个INT，这张表中的成员就是如下结构体，这个结构体中就只有一个联合体，它的宽度就是4字节：

INT里有很多个这种结构体，它与导入表一样，当遇见与该结构体相同宽度的一段0x00填充的数据则该表就结束了。同样，在这张表中发现结构体的数量，就表示你依赖该模块的函数数量。

在解析PE文件的导入表时，对于每个_IMAGE_THUNK_DATA32条目，需要按照以下逻辑处理：

1.判断导入方式，首先检查_IMAGE_THUNK_DATA32值的最高位(MSB)：

  • 如果最高位为1：表示通过序号导入。去除最高位后，剩下的31位值就是函数的导出序号，直接使用这个序号在目标DLL的导出表中查找函数
  • 如果最高位为0：表示通过名称导入。该值是一个RVA(相对虚拟地址)，指向_IMAGE_IMPORT_BY_NAME结构

2.解析_IMAGE_IMPORT_BY_NAME结构。当通过名称导入时，该结构包含以下信息：

具体解析要点：

  • Hint字段（2字节）：可能为0（表示无提示）。如果不为0，它是函数在导出表中的建议索引（可加速查找，但不保证正确）
  • Name字段：虽然声明为CHAR Name[1]，但实际上是一个以NULL结尾的ASCII字符串，需要从该位置开始读取字符，直到遇到0x00字节为止，这个完整字符串才是要导入的函数名称

这里注意在PE加载到内存前后导入表会发生如下变化：

在PE文件加载前：IAT（Import Address Table）中存储的是_IMAGE_THUNK_DATA32结构（这些条目可能是函数名称的RVA或函数的导出序号）

在PE文件加载后：PE加载器会解析所有导入函数，将实际的函数内存地址直接写入IAT中的对应位置，此时IAT中的每个条目不再是描述信息，而是真正的函数指针

事实上我们在使用依赖模块的函数时，实际上是间接调用，调用MessageBox，汇编指令去调用并不是直接call地址，而是间接的，从内存中找到地址再去call。

虽然最终只有IAT会被修改为包含实际地址，但PE文件仍然保留了两个表：

  1. IAT（Import Address Table）：最终会被修改为包含实际函数地址
  2. INT（Import Name Table，即OriginalFirstThunk指向的表）：始终保持原始导入信息

问题来了为什么不只留一张表呢，为什么需要双表设计？

可能基于以下逻辑：

  • 原始信息保留：INT始终保持原始的导入信息（函数名/序号），这在调试、反汇编或运行时查询时非常有用
  • 卸载与重载支持：当DLL需要卸载时，系统可以根据INT重新构建导入信息，动态加载/卸载DLL时不需要重新解析原始PE文件
  • 安全性验证：可以比较INT和IAT来检测IAT是否被恶意篡改
  • 延迟加载优化：可以先填充部分常用函数的地址，其他函数等到实际使用时再解析

重定位表

重定位表是PE文件中处理地址修正的关键数据结构，它使得可执行模块能够被加载到内存任意位置（地址空间布局随机化/ASLR的核心支持机制）。

为什么需要重定位：

  1. 地址空间冲突：当多个DLL都声明相同的首选基地址时
  2. ASLR保护：现代系统的地址空间布局随机化安全机制
  3. 内存管理：系统需要灵活安排模块加载位置

典型场景：一个DLL声明ImageBase为0x10000000，但该地址已被其他模块占用，系统将其加载到0x30000000，此时所有绝对地址都需要调整+0x20000000。

重定位表位置：位于PE头的DataDirectory[5]（IMAGE_DIRECTORY_ENTRY_BASERELOC），由多个IMAGE_BASE_RELOCATION块组成链式结构

核心数据结构

重定位表中有一个结构体，它一共有8字节：

但是完整的重定位表还包含了很多重定位项，所以整体大小是没有一个统一标准的，需要根据实际情况。该结构体的第一个成员VirtualAddress表示基地址，第二个成员SizeOfBlock也就是减去当前结构体的大小之后其他的所有重定位项加起来的大小。这也就表示每一个重定位表是IMAGE_BASE_RELOCATION结构体开始，跟着的就是重定位项。

从理论上来说，重定位表中存储的项应该都是4字节大小的地址，但是这样一旦需要修改的地址多了，就会占用大量的空间，所以重定位表就做了一些优化，假设你现在有这几个地址需要修正。

重定位表是按照一个物理页（4KB）进行存储的，也就表示一个4KB内存有需要修正的位置，就会有一个重定位块，一个重定位表只管自己当前的物理页的重定位。

但需要注意的是由于内存对齐的缘故，在重定位表中还是有很多的无用项的，所以需要判断当前重定位项（2字节）的高四位是否为3，如果是那么低12位就是偏移量，最后的地址也就是VirtualAddress+低12位，如果不是就表示这是无所谓的值。

PE节表

PE节表是PE文件结构中定义各个节（section）属性的关键数据结构，位于PE头（IMAGE_NT_HEADERS）之后，负责描述PE文件中每个节在磁盘和内存中的布局特征。

每个节表条目是一个40字节的固定结构（IMAGE_SECTION_HEADER），定义如下：

PE加载器根据节表信息完成从磁盘到内存的映射：

  1. 计算内存占用：VirtualAddress + max(VirtualSize, SizeOfRawData)
  2. 设置内存保护：根据Characteristics设置页属性
  3. 处理差异情况：
    • 当VirtualSize > SizeOfRawData：内存中多出部分填零
    • 当SizeOfRawData > VirtualSize：截断使用（罕见）

节数据

节数据是PE文件中存储实际内容的二进制数据块，每个节对应一个特定类型的数据，由节表（Section Table）中的条目描述其位置和属性。

  • 物理存储：节数据是PE文件中紧接在节表之后的部分，每个节数据在磁盘文件中的位置由PointerToRawData字段指定，大小由SizeOfRawData字段决定（按FileAlignment对齐）
  • 内存映射：加载到内存后，节数据的位置由VirtualAddress指定，内存中大小可能不同（由VirtualSize决定）
