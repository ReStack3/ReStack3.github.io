分布式理论

CAP 理论

CAP 理论由计算机科学家 Eric Brewer 提出，指出在分布式系统中，以下三个特性无法同时满足，最多只能满足其中两个：

  1. Consistency（一致性）：
    • 所有节点在同一时间看到的数据是一致的。
    • 例如：用户读取数据时，无论访问哪个节点，都能获取到最新的数据。
  2. Availability（可用性）：
    • 每个请求都能得到响应，即使部分节点故障。
    • 例如：系统在部分节点宕机的情况下，仍然能够提供服务。
  3. Partition Tolerance（分区容错性）：
    • 系统在遇到网络分区（节点之间无法通信）时，仍然能够继续运行。是分布式系统必须满足的特性，因为网络分区是不可避免的。
    • 例如：即使网络断开，系统仍然能够提供服务。

当网络分区(如网络抖动、节点故障)发生时，实际的选择是在 一致性（C） 和 可用性（A） 之间进行权衡。：

  • 如果选择 一致性（C），系统需要禁止部分节点的读写操作，以确保数据一致，但这会降低可用性。
  • 如果选择 可用性（A），系统需要允许所有节点的读写操作，但这可能导致数据不一致。

在网络分区未发生时（系统在绝大部分时候所处的状态），一致性和可用性是可以同时满足的。

  • 例如：在分布式数据库中，如果网络正常，所有节点可以实时同步数据，同时保证一致性和可用性。

所以CAP 理论的核心 并不是说在任何情况下只能满足两个特性，而是 在网络分区发生时，必须在一致性和可用性之间做出选择。

CAP 的权衡

  • CA（一致性 + 可用性）：
    • 放弃分区容错性，适合单机或局域网环境。
    • 例如：传统的关系型数据库（如 MySQL）在单机模式下是 CA 系统。
  • CP（一致性 + 分区容错性）：
    • 放弃可用性，适合对一致性要求高的场景。
    • 例如：ZooKeeper、Etcd 等分布式协调服务。
  • AP（可用性 + 分区容错性）：
    • 放弃一致性，适合对可用性要求高的场景。
    • 例如：Cassandra、Dynamo 等 NoSQL 数据库。

BASE理论

BASE 理论是对 CAP 理论的扩展，强调在分布式系统中，可以通过牺牲强一致性来获得高可用性和性能。即使无法做到强一致性，但每个应用都可以根据自身业务特点，采用适当的方式来使系统达到最终一致性。

也就是牺牲数据的一致性来满足系统的高可用性，系统中一部分数据不可用或者不一致时，仍需要保持系统整体“主要可用”。

  1. Basically Available（基本可用）：
    • 系统在出现故障时，仍然能够提供基本的功能。
    • 例如：电商网站在大促期间，可能会降级部分功能（如商品详情页简化），但核心功能（如下单）仍然可用。
  2. Soft State（软状态）：
    • 系统的状态可能会随时间变化，即使没有外部输入。
    • 例如：缓存数据可能会过期，需要定期更新。
  3. Eventually Consistent（最终一致性）：
    • 系统在经过一段时间后，最终会达到一致状态。
    • 例如：用户下单后，库存数据可能会延迟几秒才更新，但最终会保持一致。

BASE 理论本质上是对 CAP 的延伸和补充，更具体地说，是对 CAP 中 AP 方案的一个补充。

相较于CAP理论，BASE 的特点在于：

  • 弱一致性：允许系统在短时间内处于不一致状态。
  • 高可用性：通过牺牲强一致性，提高系统的可用性和性能。
  • 适合大规模分布式系统：例如电商、社交网络等对实时一致性要求不高的场景。

如果系统没有发生“分区”的话，节点间的网络连接通信正常的话，也就不存在 P 了。这个时候，我们就可以同时保证 C 和 A 了。因此，如果系统发生“分区”，我们要考虑选择 CP 还是 AP。如果系统没有发生“分区”的话，我们要思考如何保证 CA 。

因此，AP 方案只是在系统发生分区的时候放弃一致性，而不是永远放弃一致性。在分区故障恢复后，系统应该达到最终一致性。这一点其实就是 BASE 理论延伸的地方。
