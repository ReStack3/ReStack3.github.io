什么是Fragment？

5.1 Fragment基本概述 | 菜鸟教程

Android：Fragment详细介绍 & 使用方法解析_android fragment-CSDN博客

在 Android 中，Fragment 是一种用于构建灵活和模块化 UI 的组件。它可以被视为 Activity 的一部分，拥有自己的生命周期、布局和行为，但必须嵌入到 Activity 中才能运行。Fragment 的主要目的是支持更复杂的 UI 设计，尤其是在不同屏幕尺寸和设备上提供更好的用户体验。

Fragment 核心概念

  • Fragment 允许开发者将 UI 分解为独立的模块。每个 Fragment 可以独立开发、测试和维护，然后组合到 Activity 中。

  • Fragment 有自己的生命周期，但与 Activity 的生命周期紧密关联。Fragment 的生命周期方法（如 onCreate()、onStart()、onResume() 等）会在 Activity 的生命周期方法中被调用。

  • Fragment 可以在多个 Activity 中复用，从而减少代码重复。

  • Fragment 可以在运行时动态添加、替换或移除，这使得 UI 可以根据用户交互或设备配置动态调整。

    • [逆向与风控, Android平台, 开发, View机制]

Fragment 生命周期

Fragment 的生命周期与 Activity 的生命周期类似，但更加复杂。

onDetach() Fragment 与 Activity 解除关联时调用。 清理与 Activity 的关联，释放 Activity 引用。

Fragment 使用场景

  1. 多窗格布局：
    • 在平板或大屏设备上，Fragment 可以用于实现多窗格布局（如左侧显示列表，右侧显示详情）。
  2. 动态 UI 更新：
    • 根据用户交互动态替换或更新 UI 的一部分。
  3. 导航和分页：
    • 结合 ViewPager 或 Navigation Component 实现分页或导航功能。
  4. 复用 UI 组件：
    • 将通用的 UI 组件封装为 Fragment，以便在多个 Activity 中复用。

Android的UI管理系统的层级关系

  • PhoneWindow：管理窗口的显示和事件响应，是Activity与View系统交互的接口。
  • DecorView：是一个应用窗口的根容器，它本质上是一个 FrameLayout。DecorView 有唯一一个子 View，它是一个垂直 LinearLayout，包含两个子元素，一个是 TitleView（ActionBar的容器），另一个是 ContentView（窗口内容的容器）。关于 ContentView，它是一个 FrameLayout（android.R.id.content)，我们平常用的 setContentView 就是设置它的子 View。上图还表达了每个 Activity 都与一个 Window（具体来说是PhoneWindow）相关联，用户界面则由 Window 所承载。View 层的事件传递也都要先经过 DecorView ,然后才传递给我们的 View。
  • ViewRootImpl：连接WindowManager和DecorView，负责管理UI的渲染、布局和事件分发。(ViewRoot是早期Android版本中的类，现已被ViewRootImpl取代。)
  • ActivityThread：在Activity启动时，负责创建PhoneWindow、DecorView和ViewRootImpl，并将它们关联起来。

View绘制从ViewRoot的performTraversals方法开始，经过measure、layout、draw三个过程，然后将View绘制出来

  • measure方法用于测量View的宽高
  • layout方法用于确定View在父容器中的位置
  • draw方法负责将View绘制在屏幕上

每个流程（measure、layout、draw）都是从**顶级View（即DecorView）**开始，逐步传递到**子View**，直到完成整个View树的遍历。

这种传递机制是通过递归调用实现的：

  • 父容器调用子View的对应方法。
  • 子View再调用自己的子View的对应方法，依此类推。

以 measure 流程为例：

performMeasure：

  • 在performTraversals中调用performMeasure，开始测量流程。
  • performMeasure会调用顶级View（DecorView）的measure方法。

measure：

  • measure方法会调用onMeasure方法，执行实际的测量逻辑。

onMeasure：

  • 在onMeasure方法中，父容器（如ViewGroup）会遍历所有子View，并调用每个子View的measure方法。
  • 子View的measure方法又会调用自己的onMeasure方法，继续测量自己的子View。
  • 这样，测量流程就从父容器传递到子View，再从子View传递到子View的子View，直到完成整个View树的测量。

理解 MeasureSpec

MeasureSpec 是 Android 中用于在 View 的测量过程中传递尺寸信息的工具。它封装了父容器对子 View 的尺寸要求，帮助子 View 确定自己的测量宽高。MeasureSpec 是一个 32 位的整型值，由两部分组成：

  1. 模式（Mode）：高 2 位，表示父容器对子 View 的尺寸约束模式。
  2. 尺寸（Size）：低 30 位，表示父容器对子 View 的具体尺寸要求。

MeasureSpec 的三种模式

MeasureSpec 定义了三种模式，用于描述父容器对子 View 的尺寸约束：

模式 值 描述
UNSPECIFIED 0 父容器对子 View 没有限制，子 View 可以想要多大就多大（通常用于 ScrollView 等可滚动的容器）。
EXACTLY 1 父容器已经确定了子 View 的精确尺寸，子 View 必须使用这个尺寸。
AT_MOST 2 子 View 的尺寸不能超过父容器指定的最大尺寸，但可以小于这个尺寸。

MeasureSpec 的使用场景

在 View 的测量过程中，父容器会根据自己的尺寸和布局规则，生成子 View 的 MeasureSpec，并通过 measure 方法传递给子 View。子 View 根据 MeasureSpec 的约束，确定自己的测量宽高。

父容器生成 MeasureSpec 的规则

  • 父容器的 MeasureSpec 和子 View 的 LayoutParams 共同决定了子 View 的 MeasureSpec。
  • 例如：
    • 如果子 View 的 LayoutParams 是 MATCH_PARENT，父容器的模式是 EXACTLY，则子 View 的 MeasureSpec 也是 EXACTLY，尺寸等于父容器的剩余空间。
    • 如果子 View 的 LayoutParams 是 WRAP_CONTENT，父容器的模式是 AT_MOST，则子 View 的 MeasureSpec 也是 AT_MOST，尺寸不能超过父容器的剩余空间。

子 View 处理 MeasureSpec

  • 子 View 在 onMeasure 方法中根据 MeasureSpec 的约束，计算自己的测量宽高。
  • 例如：
    • 如果模式是 EXACTLY，子 View 必须使用给定的尺寸。
    • 如果模式是 AT_MOST，子 View 的尺寸不能超过给定的尺寸。
    • 如果模式是 UNSPECIFIED，子 View 可以根据自己的内容决定尺寸。

View的工作流程

具体见：Android View原理浅析——View的工作原理_android viewroot-CSDN博客

View 的工作流程是 Android 中 UI 渲染的核心过程，主要包括 测量（measure）、布局（layout） 和 绘制（draw） 三个阶段。这些流程由 ViewRootImpl 触发，从顶级 View（通常是 DecorView）开始，递归遍历整个 View 树，最终完成 UI 的显示。以下是 View 工作流程的详细解析：

详细流程

(1) Measure 阶段

  • 作用：测量 View 的宽高，确定其尺寸。
  • 流程：
    1. ViewRootImpl#performTraversals 调用 performMeasure。
    2. performMeasure 调用顶级 View（DecorView）的 measure 方法。
    3. measure 方法调用 onMeasure 方法。
    4. 在 onMeasure 中：
      • View 根据父容器传递的 MeasureSpec 计算自己的宽高。
      • 如果是 ViewGroup，会遍历所有子 View，并调用子 View 的 measure 方法。
    5. 递归完成后，View 的 mMeasuredWidth 和 mMeasuredHeight 被赋值。
  • 关键点：
    • MeasureSpec 决定了 View 的测量模式（EXACTLY、AT_MOST、UNSPECIFIED）。
    • 测量结果通过 setMeasuredDimension 方法设置。

(2) Layout 阶段

  • 作用：确定 View 在父容器中的位置（四个顶点的坐标）。
  • 流程：
    1. ViewRootImpl#performTraversals 调用 performLayout。
    2. performLayout 调用顶级 View（DecorView）的 layout 方法。
    3. layout 方法调用 onLayout 方法。
    4. 在 onLayout 中：
      • View 根据测量结果确定自己的位置（mLeft、mTop、mRight、mBottom）。
      • 如果是 ViewGroup，会遍历所有子 View，并调用子 View 的 layout 方法。
  • 关键点：
    • 布局完成后，View 的位置信息被确定。
    • View 的实际宽高通常等于测量宽高。

(3) Draw 阶段

  • 作用：将 View 的内容绘制到屏幕上。
  • 流程：
    1. ViewRootImpl#performTraversals 调用 performDraw。
    2. performDraw 调用顶级 View（DecorView）的 draw 方法。
    3. draw 方法依次执行以下步骤：
      • 绘制背景（drawBackground）。
      • 绘制内容（onDraw）。
      • 绘制子 View（dispatchDraw）。
      • 绘制装饰（如滚动条、前景等）。
    4. 如果是 ViewGroup，会通过 dispatchDraw 方法遍历所有子 View，并调用子 View 的 draw 方法。
  • 关键点：
    • 绘制完成后，View 的内容会显示在屏幕上。
    • View 的绘制顺序是：背景 → 内容 → 子 View → 装饰。
    • [逆向与风控, Android平台, 开发, View机制]

View 工作流程的触发时机

  • 首次绘制：
    • 当 Activity 启动时，ViewRootImpl 会触发第一次 performTraversals，完成 View 树的测量、布局和绘制。
  • 更新 UI：
    • 当 View 的状态发生变化（如尺寸、位置、内容等），ViewRootImpl 会再次触发 performTraversals，重新执行测量、布局和绘制。
  • 强制刷新：
    • 调用 View#invalidate 或 View#requestLayout 可以强制触发 View 的重新绘制或重新布局。

理解RemoteViews

参考：Android 理解RemoteViews本章先简单介绍通知栏和桌面小部件应用，接着分析RemoteViews内部机制， - 掘金

android进阶（五）—–理解RemoteViews一、RemoteView的应用 RemoteView主要用在 - 掘金

RemoteViews 是 Android 提供的一种机制，用于跨进程更新 UI。它主要用于以下场景：

  1. App Widgets：在桌面小部件（App Widgets）中更新 UI。
  2. 通知栏：在通知栏中自定义通知的 UI。
  3. 跨进程 UI 更新：在某些情况下，一个进程需要更新另一个进程的 UI。

RemoteViews 的核心思想是将 UI 的描述和操作封装成一个对象，然后通过跨进程通信（IPC）机制传递给目标进程，由目标进程根据描述更新 UI。

  • [逆向与风控, Android平台, 开发, View机制]

RemoteViews 的内部机制

1. 描述 UI 结构

RemoteViews 通过 XML 布局文件或代码动态构建的方式描述 UI 结构。它并不是直接操作 View，而是将 UI 的描述信息封装在 RemoteViews 对象中。

2. 跨进程传递

RemoteViews 实现了 Parcelable 接口，因此可以通过 Binder 跨进程传递。当 RemoteViews 被传递到目标进程时，目标进程会根据描述信息重新构建 UI。

3. 更新 UI

RemoteViews 通过 apply() 方法将 UI 描述应用到目标进程中。apply() 方法会创建一个新的 View 层次结构，并将其附加到目标容器中。

4. 操作 UI

RemoteViews 提供了一些方法（如 setTextViewText()、setImageViewResource() 等）来更新 UI 元素。这些操作会被封装为 Action 对象，并在目标进程中执行。

RemoteViews 的核心实现

1. RemoteViews 的组成

  • 布局 ID：描述 UI 的布局文件 ID。
  • Action 列表：记录对 UI 的操作（如设置文本、设置图片等）。
  • 应用包名：用于加载目标进程的资源。

2. Action 机制

RemoteViews 中的每一个 UI 操作（如设置文本、设置图片等）都会被封装为一个 Action 对象。Action 是一个抽象类，具体的操作由其子类实现，例如：

  • TextViewSizeAction：设置文本大小。
  • SetDrawableParameters：设置图片参数。
  • ReflectionAction：通过反射调用 View 的方法。

当 RemoteViews 被传递到目标进程时，这些 Action 对象会被依次执行，从而更新 UI。

3. 跨进程通信
