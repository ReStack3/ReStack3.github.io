View基础知识

什么是View?

View是Android中所有控件的基类，代表界面层的控件抽象。ViewGroup是View的子类，用于包含多个控件，形成控件组。

继承关系：ViewGroup继承自View，因此View可以是单个控件或一组控件。

View树结构：View和ViewGroup的嵌套关系形成类似Web前端DOM树的View树结构。

View的位置参数

  1. Left：View左边到其父布局左边的距离（单位：像素）。
  2. Top：View顶部到其父布局顶部的距离（单位：像素）。
  3. Right：View右边到其父布局左边的距离（单位：像素）。
  4. Bottom：View底部到其父布局顶部的距离（单位：像素）。

此外，View还提供了以下方法获取位置信息：

  • getX()：获取View的X坐标（相对于父布局）。
  • getY()：获取View的Y坐标（相对于父布局）。
  • getTranslationX()：获取View在X轴上的偏移量。
  • getTranslationY()：获取View在Y轴上的偏移量。

其中x和y是View左上角的坐标，而translationX和translationY是View左上角相对于父容器的偏移量

注意：View的位置参数是基于其父布局的坐标系，而非屏幕坐标系。

MotionEvent和TouchSlop

MotionEvent

MotionEvent 是 Android 中用于描述触摸事件的类，封装了用户触摸屏幕时的所有信息，包括触摸的位置、动作类型、时间等。

常用动作类型

  • ACTION_DOWN：手指按下屏幕时触发。
  • ACTION_MOVE：手指在屏幕上移动时触发。
  • ACTION_UP：手指离开屏幕时触发。
  • ACTION_CANCEL：触摸事件被取消时触发（例如父布局拦截了事件）。

常用方法

  • getX()：获取触摸点相对于当前 View 的 X 坐标。
  • getY()：获取触摸点相对于当前 View 的 Y 坐标。
  • getRawX()：获取触摸点相对于屏幕的 X 坐标。
  • getRawY()：获取触摸点相对于屏幕的 Y 坐标。
  • getAction()：获取当前事件的动作类型。
  • getPointerCount()：获取触摸屏幕的手指数量（用于多点触控）。

TouchSlop

TouchSlop 是 Android 系统定义的一个常量，表示系统认为的“最小滑动距离”。当用户手指在屏幕上移动的距离小于 TouchSlop 时，系统不会将其视为滑动事件，而是认为用户只是轻微抖动或点击。

View的滑动

View的滑动在Android开发中至关重要，无论是下拉刷新、侧滑菜单还是其他交互效果，滑动都是实现这些功能的基础。由于Android设备的屏幕尺寸有限，滑动能够帮助隐藏和显示更多内容，从而提升用户体验。掌握滑动的方法是实现复杂自定义控件的基础。常见的View滑动方式主要有三种：

  1. 使用View的scrollTo/scrollBy方法：通过直接调用View的这两个方法实现滑动，适合简单的滑动需求，但需要注意坐标系的特点。
  2. 使用动画实现滑动：通过平移动画（如属性动画或补间动画）让View产生滑动效果，适合实现平滑的过渡效果。
  3. 改变View的LayoutParams：通过动态修改View的布局参数（如margin或width等）使其重新布局，从而实现滑动，适合需要精确控制布局的场景。

使用scrollTo/scrollBy

scrollTo 和 scrollBy 是 View 提供的用于实现滑动的方法，它们通过改变 View 的内容的偏移量来实现滑动效果。需要注意的是，这两种方法滑动的是 View 的内容，而不是 View 本身。

滑动的是内容：scrollTo 和 scrollBy 滑动的是 View 的内容，而不是 View 本身。例如，如果 View 是一个容器（如 LinearLayout），滑动的是其内部子 View 的位置。

坐标系方向：

mScrollX：表示 View 左边缘与 View 内容左边缘在水平方向的距离。

mScrollY：表示 View 上边缘与 View 内容上边缘在竖直方向的距离。

  • x 和 y 的值可以为正或负。
  • 向右滑动：x 为负值。
  • 向左滑动：x 为正值。
  • 向下滑动：y 为负值。
  • 向上滑动：y 为正值。

无动画效果：滑动是瞬间完成的，没有平滑过渡效果。

注意事项:

  • 滑动范围：scrollTo 和 scrollBy 的滑动范围受 View 的内容大小限制。如果内容不足以滑动，则不会产生效果。
  • 获取当前滑动位置：可以通过 getScrollX() 和 getScrollY() 获取当前 View 内容的滑动偏移量。

使用动画

使用动画实现 View 的滑动是一种常见且灵活的方式，可以通过补间动画（Tween Animation）或属性动画（Property Animation）来实现。动画滑动不仅能够实现平滑的过渡效果，还可以控制滑动的速度、加速度等属性。

各种滑动方式的对比

弹性滑动

知道了View的滑动，我们还要知道如何实现View的弹性滑动，比较生硬地滑动过去，这种方式的用户体验实在太差了，因此我们要实现渐近式滑动。那么如何实现弹性滑动呢？其实实现方法有很多，但是它们都有一个共同思想：将一次大的滑动分成若干次小的滑动并在一个时间段内完成。

computeScrollOffset

该方法通常用于实现平滑的滚动或滑动动画。在 View 或自定义控件中，通过不断调用 computeScrollOffset 来更新滚动位置，从而实现动态效果。适用于滚动列表、滑动页面、拖拽等场景。其根据时间流逝的百分比来算出scrollX和scrollY改变的百分比并计算出当前的值，这个过程类似于动画中的插值器的概念。

invalidate方法会导致View重绘，在View的draw方法中又会去调用computeScroll方法，computeScroll方法在View中是一个空实现,需要自定义实现。这个流程如下：

当View重绘后会在draw方法中调用computeScroll，而computeScroll又会去向Scroller获取当前的scrollX和scrollY；然后通过scrollTo方法实现滑动；接着又调用postInvalidate方法来进行第二次重绘，这一次重绘的过程和第一次重绘一样，还是会导致computeScroll方法被调用；然后继续向Scroller获取当前的scrollX和scrollY，并通过scrollTo方法滑动到新的位置，如此反复，直到整个滑动过程结束。

Scroller 的工作原理

  1. Scroller 的角色：

    • Scroller 本身并不直接控制 View 的滑动，它只是一个计算工具，用于根据时间、速度和距离等参数，计算出 View 在某一时刻应该滑动到的位置。

  2. 与 View 的配合：

    • Scroller 需要与 View 的 computeScroll 方法配合使用。
    • computeScroll 是 View 的一个回调方法，当 View 需要重绘时，系统会调用该方法。

  3. 滑动过程：

    • 启动滑动：通过调用 Scroller.startScroll 或 Scroller.fling 方法，初始化滑动的起始位置、目标位置、持续时间等参数。
    • 重绘触发滑动：在 computeScroll 方法中，调用 Scroller.computeScrollOffset 计算当前时刻 View 应该滑动到的位置。
    • 更新位置：通过 View.scrollTo 或 View.scrollBy 方法，将 View 滑动到计算出的位置。
    • 循环重绘：Scroller 会不断触发 View 的重绘，每次重绘都会调用 computeScroll，从而更新 View 的位置，直到滑动完成。

  4. 弹性滑动的实现：

    • Scroller 通过插值器（Interpolator）或样条曲线（Spline）来计算滑动过程中的位置变化，从而实现平滑的滑动效果。
    • 每一次重绘都会导致 View 进行小幅度的滑动，多次小幅度滑动组合在一起，就形成了弹性滑动的效果。

  5. 滑动结束：

    • 当滑动时间到达总持续时间（mDuration），或者滑动位置到达目标位置时，Scroller 会标记滑动完成（mFinished = true），此时 computeScroll 返回 false，停止重绘。

通过动画

通过动画实现 View 的弹性滑动，是一种非常灵活且强大的方式。它的核心思想是利用动画的逐帧回调机制，在每一帧中根据动画的进度计算 View 的滑动距离，并通过 scrollTo 方法更新 View 的内容位置。与 Scroller 相比，动画实现方式更加灵活，适用于更复杂的动画场景。

  • [逆向与风控, Android平台, 开发, View机制]

利用动画实现弹性滑动的原理

  1. 动画的逐帧回调：

    • 动画的本质是一个渐进的过程，系统会在每一帧中回调 onAnimationUpdate 方法。
    • 在 onAnimationUpdate 中，我们可以获取动画的当前进度（百分比），并根据这个进度计算 View 的滑动距离。

  2. 滑动距离的计算：

    • 根据动画的进度（fraction），计算 View 的当前滑动位置。
    • 例如，如果目标滑动距离是 100px，动画进度是 0.5，则当前滑动距离为 50px。

  3. 更新 View 的位置：

    • 通过 scrollTo 方法，将 View 的内容滑动到计算出的位置。

  4. 灵活性和扩展性：

    • 除了实现弹性滑动，还可以在 onAnimationUpdate 中添加其他操作，比如缩放、旋转等，从而实现更复杂的动画效果。

使用延时策略

延时策略是另一种实现弹性滑动的有效方法，其核心思想是通过发送一系列延时消息，将滑动过程分解为多个小步骤，逐步更新 View 的位置，从而实现平滑的滑动效果。这种方式与动画和 Scroller 相比，更加底层，但也更加灵活。下面我们详细分析延时策略的实现原理，并通过代码示例说明其具体实现。

  • [逆向与风控, Android平台, 开发, View机制]

延时策略的核心思想

  1. 逐帧更新：

    • 将滑动过程分解为多个小步骤（帧），每一帧中 View 滑动一小段距离。
    • 通过延时机制（如 Handler.postDelayed 或 Thread.sleep），控制每一帧的时间间隔。

  2. 滑动距离的计算：

    • 根据当前帧的进度（fraction），计算 View 的滑动距离。
    • 例如，如果目标滑动距离是 100px，总帧数为 30，则每一帧滑动 100 / 30 ≈ 3.33px。

  3. 延时机制：

    • 使用 Handler.postDelayed 或 Thread.sleep，在每一帧之间添加延时，从而控制滑动的速度。

  4. 灵活性和局限性：

    • 延时策略非常灵活，可以精确控制每一帧的滑动距离和时间间隔。
    • 但由于系统消息调度的不确定性，延时时间无法完全精确。

View的事件分发机制

View 的事件分发机制是 Android 系统中处理用户交互事件（如触摸事件、按键事件等）的核心机制。它的主要作用是将事件从顶级 View（DecorView）逐级传递到具体的子 View，并在传递过程中决定哪个 View 应该处理该事件。理解事件分发机制对于开发复杂的用户交互逻辑至关重要。

点击事件的传递规则

事件分发机制的核心概念

  1. 事件类型：

    • 触摸事件（TouchEvent）：包括 ACTION_DOWN、ACTION_MOVE、ACTION_UP 等。
    • 按键事件（KeyEvent）：包括 KEY_DOWN、KEY_UP 等。
    • 其他事件：如 ACTION_CANCEL、ACTION_OUTSIDE 等。

  2. 事件分发流程：

    • 事件从顶级 View（通常是 DecorView）开始，逐级向下传递到具体的子 View。
    • 在传递过程中，每个 View 都可以决定是否拦截事件、是否消费事件，以及是否继续传递事件。

  3. 核心方法：

    • **dispatchTouchEvent**：负责事件的分发。
    • **onInterceptTouchEvent**：负责事件拦截（仅 ViewGroup 有该方法）。
    • **onTouchEvent**：负责事件的处理。

事件分发流程

事件分发机制的整体流程可以分为以下几个步骤：

  1. 事件从 Activity 开始：

    • 事件首先传递给 Activity 的 dispatchTouchEvent 方法。

  2. 事件传递给顶级 View（DecorView）：

    • Activity 将事件传递给 DecorView 的 dispatchTouchEvent 方法。

  3. 事件在 ViewGroup 中传递：

    • 在 ViewGroup 中，事件会先调用 onInterceptTouchEvent 方法，判断是否拦截事件。
      • 如果拦截，则事件不会继续向下传递，而是由该 ViewGroup 的 onTouchEvent 方法处理。
      • 如果不拦截，则事件会继续传递给子 View。

  4. 事件在子 View 中处理：

    • 如果事件传递到子 View，子 View 的 dispatchTouchEvent 方法会被调用。
    • 子 View 可以通过 onTouchEvent 方法决定是否消费事件。
      • 如果消费事件，则事件分发流程结束。
      • 如果不消费事件，则事件会回传给父 View 的 onTouchEvent 方法。

  5. 事件回传：

    • 如果事件没有被任何 View 消费，最终会回传给 Activity 的 onTouchEvent 方法。

事件分发流程示例

以下是一个典型的事件分发流程示例：

  1. 用户点击屏幕，触发 ACTION_DOWN 事件。
  2. 事件传递给 Activity 的 dispatchTouchEvent 方法。
  3. Activity 将事件传递给 DecorView 的 dispatchTouchEvent 方法。
  4. DecorView 将事件传递给其子 ViewGroup 的 dispatchTouchEvent 方法。
  5. ViewGroup 调用 onInterceptTouchEvent 方法，判断是否拦截事件。
    • 如果拦截，则事件由 ViewGroup 的 onTouchEvent 方法处理。
    • 如果不拦截，则事件继续传递给子 View。
  6. 子 View 调用 dispatchTouchEvent 方法，并尝试处理事件。
    • 如果子 View 的 onTouchEvent 方法返回 true，则事件被消费，分发流程结束。
    • 如果返回 false，则事件回传给父 ViewGroup 的 onTouchEvent 方法。
  7. 如果事件未被任何 View 消费，最终会回传给 Activity 的 onTouchEvent 方法。

假如点击事件是一个难题，这个难题最终被上级领导分给了一个程序员去处理（这是事件分发过程），结果这个程序员搞不定（onTouchEvent返回了false），现在该怎么办呢？难题必须要解决，那只能交给水平更高的上级解决（上级的onTouchEvent被调用），如果上级再搞不定，那只能交给上级的上级去解决，就这样将难题一层层地向上抛.

以下是关于 Android 事件传递机制的总结：

编号 结论 说明
1 同一个事件序列是指从手指接触屏幕的那一刻起，到手指离开屏幕的那一刻结束，包括 ACTION_DOWN、ACTION_MOVE 和 ACTION_UP。 事件序列以 ACTION_DOWN 开始，中间可能包含多个 ACTION_MOVE，最终以 ACTION_UP 结束。
2 正常情况下，一个事件序列只能被一个 View 拦截且消耗。 一旦某个 View 拦截了事件，整个事件序列都会交给它处理。特殊情况下，可以通过 onTouchEvent 强行将事件传递给其他 View。
3 某个 View 一旦决定拦截，那么整个事件序列都只能由它处理，并且它的 onInterceptTouchEvent 不会再被调用。 拦截后，系统会直接将事件交给该 View 处理，不再询问是否拦截。
4 某个 View 一旦开始处理事件，如果它不消耗 ACTION_DOWN 事件（onTouchEvent 返回 false），那么同一事件序列中的其他事件都不会再交给它处理，事件将重新交由它的父元素处理。 如果 View 不消耗 ACTION_DOWN，父元素的 onTouchEvent 会被调用。
5 如果 View 不消耗除 ACTION_DOWN 以外的其他事件，那么这个点击事件会消失，父元素的 onTouchEvent 并不会被调用，当前 View 可以持续收到后续事件，最终这些消失的事件会传递给 Activity 处理。 不消耗事件时，事件会消失，并最终传递给 Activity。
6 ViewGroup 默认不拦截任何事件。Android 源码中 ViewGroup.onInterceptTouchEvent 方法默认返回 false。 ViewGroup 默认不拦截事件，事件会传递给子 View。
7 View 没有 onInterceptTouchEvent 方法，一旦有点击事件传递给它，那么它的 onTouchEvent 方法就会被调用。 View 只能通过 onTouchEvent 处理事件，无法拦截事件。
8 View 的 onTouchEvent 默认都会消耗事件（返回 true），除非它是不可点击的（clickable 和 longClickable 同时为 false）。 Button 的 clickable 默认为 true，TextView 的 clickable 默认为 false。
9 View 的 enable 属性不影响 onTouchEvent 的默认返回值。哪怕一个 View 是 disable 状态的，只要它的 clickable 或者 longClickable 有一个为 true，那么它的 onTouchEvent 就返回 true。 enable 属性只影响 View 的可交互状态，不影响事件消耗。
10 onClick 会发生的前提是当前 View 是可点击的，并且它收到了 ACTION_DOWN 和 ACTION_UP 的事件。 onClick 是一个完整点击事件的回调，需要 ACTION_DOWN 和 ACTION_UP。
11 事件传递过程是由外向内的，即事件总是先传递给父元素，然后再由父元素分发给子 View。通过 requestDisallowInterceptTouchEvent 方法可以在子元素中干预父元素的事件分发过程，但是 ACTION_DOWN 事件除外。 子 View 可以通过 requestDisallowInterceptTouchEvent 阻止父 ViewGroup 拦截事件，但 ACTION_DOWN 事件无法干预。

View的滑动冲突

常见的滑动冲突场景

外部左右滑动，内部上下滑动

典型场景：

  • ViewPager + Fragment，每个 Fragment 内部是一个 ListView 或 RecyclerView。
  • 外部通过左右滑动切换页面，内部通过上下滑动滚动内容。

冲突原因：

  • 当用户斜向滑动时，系统无法确定是应该触发外部左右滑动，还是内部上下滑动。

外部和内部同方向滑动

典型场景：

  • 外部是 ScrollView，内部是 RecyclerView，两者都支持上下滑动。
  • 外部是 HorizontalScrollView，内部是 ViewPager，两者都支持左右滑动。

冲突原因：

  • 当用户滑动时，系统无法确定是应该让外部控件滑动，还是让内部控件滑动。
  • 导致的结果是：
    • 只有一层能够滑动。
    • 或者两层都滑动，但滑动效果卡顿。
    • [逆向与风控, Android平台, 开发, View机制]

多层嵌套滑动

典型场景：

  • 外部是 SlideMenu（支持左右滑动），内部是 ViewPager（支持左右滑动），ViewPager 的每个页面又是一个 ListView（支持上下滑动）。
  • 外部是 CoordinatorLayout，内部是 RecyclerView，RecyclerView 的每个 Item 又是一个 ViewPager。

冲突原因：

  • 多层嵌套滑动导致事件分发更加复杂。
  • 例如，用户斜向滑动时，系统需要同时处理外部、中层和内层的滑动逻辑。

常见场景：

场景 冲突描述 解决方案
场景 1：ViewPager + RecyclerView ViewPager 支持左右滑动，RecyclerView 支持上下滑动。
当用户斜向滑动时，系统无法确定是应该触发 ViewPager 的左右滑动，还是 RecyclerView 的上下滑动。		 ViewPager 内部已经处理了这种冲突，开发者无需额外处理。
场景 2：ScrollView + RecyclerView ScrollView 和 RecyclerView 都支持上下滑动。
当用户滑动时，系统无法确定是应该让 ScrollView 滑动，还是让 RecyclerView 滑动。		 使用 外部拦截法 或 内部拦截法，明确滑动逻辑。
场景 3：SlideMenu + ViewPager + ListView SlideMenu 支持左右滑动，ViewPager 支持左右滑动，ListView 支持上下滑动。
当用户滑动时，系统需要同时处理多层滑动逻辑。		 分解为两个单一滑动冲突：
1. 处理 SlideMenu 和 ViewPager 之间的滑动冲突（场景 2）。
2. 处理 ViewPager 和 ListView 之间的滑动冲突（场景 1）。

滑动冲突的处理规则

场景 1：根据滑动方向（水平或竖直）决定事件由谁拦截。

场景 2：从业务需求出发，动态调整事件拦截逻辑。

场景 3：将多层嵌套滑动分解为多个单一滑动冲突，分别处理。
