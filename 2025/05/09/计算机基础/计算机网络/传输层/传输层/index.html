传输层

传输层

从通信和信息处理的角度看，传输层向它上面的应用层提供通信服务，它属于面向通信部分的最高层，同时也是用户功能中的最底层。为运行在不同主机上的进程之间提供了逻辑通信。

网络的边缘部分的两台主机使用网络核心部分的功能进行端到端的通信时，只有主机的协议栈才有传输层和应用层，而路由器在转发分组时都只用到下三层的功能（即 在通信子网中没有传输层，传输层只存在于通信子网以外的主机中）。

回顾之前的学过的知识：

物理层为数据通信提供实际的物理线路和通信信道，这是任何数据通信的基础；数据链路层为同一网络中（数据链路层的通信限于同一局域网中）的数据通信提供了虚拟的通信通道，可以根据不同链路类型对物理层的比特流进行帧封装和传输；网络层为不同网络间的数据通信提供了数据包的路由、转发功能，把数据包从一个网络中的主机传送到位于另一网络中的目的主机上，其中需要选择传送的最佳路径。

既然网络层已把源主机上发出的数据包传送给了目的主机，那么为什么还需要设置一个传输层呢？

位于两台网络主机间的真正数据通信主体不是这两台主机，而是两台主机中的各种网络应用进程。因为在同一时刻，两主机间可以进行多个应用通信。
而这里的应用进程识别就要依靠“传输层”了，它就是通过“端口”将不同应用进程进行对应的。

什么是端口呢？

我们知道单个主机上进程是用进程标识符来标志的，而在网络通信时各种应用程序却不应当让计算机操作系统指派它的进程标识符，往往需要利用目的主机提供的功能来识别终点，而不需要知道实现这个功能的进程。即端口是传输层的服务访问点(TSAP)，用来表示主机中的应用进程。

属性 说明
定义 传输层的服务访问点（TSAP），用于唯一标识主机中的应用进程。
层次对应SAP - 数据链路层：MAC地址 - 网络层：IP地址 - 传输层：端口号
端口号范围 16位整数（0~65535）
作用范围 仅本地有效，不同主机间相同端口号无关联。
核心功能 实现传输层的复用与分用，支持多进程共享网络资源。
通信必备条件 需同时指定 目标IP地址（主机） + 目标端口号（进程）。

在网络中通过IP地址来标识和区别不同的主机，通过端口号来标识和区分一台主机中的不同应用进程。在网络中采用发送方和接收方的套接字(Socket)组合来识别端点。【所谓套接字，实际上是一个通信端点，即 套接字=（主机IP地址，端口号）】

套接字的长度为48位：32位IP地址+16位端口号。它唯一地标识网络中的一台主机和其上的一个应用（进程）。进程通过套接字来接收和发送报文，套接字相当于一个通道。

给出了上述知识，就可以看到传输层的完整功能了:

传输层向高层用户屏蔽了低层网络核心的细节（如网络拓扑、路由协议等），它使应用进程看见的是好像在两个传输层实体之间有一条端到端的逻辑通信信道，这条通信信道对上层的表现却因传输层协议不同而有很大的差别。

  • TCP时，尽管下面的网络是不可靠的，当这种通信信道就相当于一条全双工的可靠信道。
  • UDP时，这种逻辑通信信道仍然是一条不可靠信道。

通过这种机制位于网络两端的不同主机之间的进程通信实现了近似同一个主机上进程间通信的效果，这种抽象也同后来的RPC机制有异曲同工之妙。

协议分析

为了满足不同应用场景对通信的 可靠性 和 效率 的差异化需求。

  • TCP 确保数据按序、完整、无丢失地传输，适用于对数据准确性要求高的场景（如网页浏览、文件传输、电子邮件）。然而，这些保障机制带来了额外的开销（如首部较大、延迟较高）。
  • UDP 则是无连接的、不可靠的协议，仅提供最基本的传输功能（无连接管理、无重传、无流量控制），但其首部更小、传输延迟更低、资源消耗更少，适用于对实时性要求高且能容忍少量丢包的应用（如视频会议、在线游戏、DNS查询）。

接下来将详细分析这两个协议的细节和差别

用户数据报协议【UDP】

什么是UDP

RFC768定义的用户数据报协议(UDP，User Datagram Protocol)只是做了传输协议能够做的最少工作，它仅在IP的数据报服务之上增加了两个最基本的服务：复用和分用以及差错检测。

特性 UDP（用户数据报协议） 对比TCP（传输控制协议）
连接方式 无连接（无需握手或断开） 面向连接（三次握手、四次挥手）
可靠性 不可靠（尽最大努力交付，无确认、重传、排序） 可靠（确认应答、超时重传、数据排序）
数据传输单位 面向报文：保留应用层报文边界，不拆分不合并，一次交付完整报文 面向字节流：数据视为无边界字节流，可能分片
拥塞控制 无拥塞控制（发送速率不受网络拥塞影响） 有拥塞控制（如慢启动、拥塞避免）
通信模式 支持一对一、一对多、多对一、多对多（如广播、多播） 仅支持一对一通信
首部开销 固定8字节（含源端口、目的端口、长度、校验和） 至少20字节（含序列号、确认号、窗口等字段）
适用场景 实时应用（视频会议、在线游戏、DNS查询）、容忍丢包但要求低延迟 需可靠传输的应用（网页浏览、文件传输、邮件）
典型协议/应用 DNS、DHCP、QUIC（HTTP/3）、SNMP、实时流媒体（RTP） HTTP、HTTPS、FTP、SMTP、SSH

UDP报文字段

  • 伪首部（Pseudo Header）

校验和计算时临时添加的12字节结构（不实际传输），包含源/目的IP地址、协议类型（UDP=17）和UDP长度，用于增强数据完整性验证。格式：

  • 源端口：2字节。源端口号。
  • 目的端口：2字节。目的端口号。
  • 长度：2字节，单位为1B。UDP数据报的长度（包括首部和数据），其最小值是8（仅有首部）。
  • 校验和：2字节。检测UDP数据报在传输中是否有错，有错就丢弃。（二进制反码运算求和再取反）

传输控制协议【TCP】

什么是TCP

传输控制协议(TCP，Transmission Control Protocol)是在不可靠的IP层之上实现的可靠的数据传输协议，它主要解决传输的可靠、有序、无丢失和不重复问题。

特性 TCP（传输控制协议）
连接方式 面向连接：需通过三次握手建立连接，四次挥手释放连接
通信模式 严格一对一（单播），每条连接仅限两个端点（源IP:Port + 目的IP:Port）
可靠性 可靠交付： • 确认应答（ACK） • 超时重传 • 数据排序 • 流量/拥塞控制
双工性 全双工通信：同一连接中可双向同时传输数据（独立维护发送/接收缓冲区）
数据传输形式 面向字节流： • 数据视为无结构的连续字节序列 • 可能合并/拆分应用层数据块
典型应用场景 HTTP网页加载、文件传输（FTP）、电子邮件（SMTP）、数据库连接

TCP连接管理

无论是三次握手还是四次挥手，都需要双方共同确认一次消息，才能算是真正约定好了。只要有一方没有真正确认此次约定，那此次约定就无法构成。

连接建立条件：

  • 要使每一方都能够确知对方的存在。
  • 要允许双方协商一些参数（如最大报文段长度MMS、最大窗口值、时间戳选项及服务质量等）
  • 能够对运输实体资源（如缓存大小，连接表中的项目等）进行分配。

TCP连接的端口称为套接字(socket)或插口。端口拼接到IP地址即构成套接字。每条TCP连接唯一地被通信两端的两个端点（即 两个套接字）确定。采用C/S方式（客户/服务器方式）（客户机是主动发起连接的应用进程，服务器是被动等待连接的应用进程）

TCP连接建立(三次握手)

  1. 第一步：客户机的TCP首先向服务器的TCP发送一个连接请求报文段。其首部中的SYN标志位被置为1（“要同步连接了”），客户机会随机选择一个起始序号字段seq=x（“我的起始序号为x”）（作为最开始的起始序号，要被消耗）。（连接请求报文不携带数据，但要消耗一个序号）
  2. 第二步：服务器的TCP收到连接请求报文段后，若同意连接，就发送一个确认报文段，并为该TCP连接分配TCP缓存和变量。SYN=1（“要同步连接了”）,ACK=1（“确认收到了连接”），确认号字段ack=x+1（“我收到你发过来的序号x的报文段了，我现在想要你的x+1”），并且服务器随机产生起始序号字段seq=y（“我的起始序号为y”）。（确认报文不携带数据，但也要消耗一个序号）
  3. 第三步：但客户机收到确认报文段后，还要向服务器给出确认收到，并且也要给该连接分配缓存和变量。ACK=1（“确认收到了”）,seq=x+1（“给你发序号x+1的报文段”）,ack=y+1（“我想要你的y+1”）。（该报文段可以携带数据，发出请求，若不携带数据则不消耗序号）

TCP连接释放(四次挥手)

  1. 第一步：客户机打算关闭连接时，向其TCP发送一个连接释放报文段，并停止发送数据，主动关闭TCP连接。FIN=1（“我发完了，要释放了”）,seq=u（它等于前面已传送过的数据的最后一个字节的序号+1）（“我现在给你发下一个报文段：u，虽然没啥可发的了，意思一下”）。（FIN报文段即使不携带数据，也要消耗一个序号）
  2. 第二步：服务器收到连接释放报文段后即发出确认。ACK=1（“确认收到了”）,seq=v（序号等于前面已传送过的数据的最后一个字节的序号+1）（“你发完了，我还没发完呢，我给你发下一个报文段：v”）,ack=u+1（“我收到你发过来的u了，我现在期望收到u+1”）。
  3. 第三步：若服务器已经没有要向客户机发送的数据，就通知TCP释放连接，发送连接释放报文段。FIN=1（“我也发完了，释放吧”）,ACK=1（“确认知道你要释放了，巧了，我也想释放”）,seq=w（“我现在给你发下一个报文段：w，虽然我也没啥可发的了，但是意思一下也不坏”）,ack=u+1（“你这u+1给我发的什么啊，怎么什么都没有，再给我发一次”）。
  4. 第四步：客户机收到连接释放报文段后，必须发出确认报文段。ACK=1（“确认收到了”）,seq=u+1（“给你给你，你要的u+1”）,ack=w+1（“我现在想要下一个w+1”）。（但是这个已经得不到回应了，服务器已经发完了）（所以客户机等了一会要不到，就关闭了）

为什么不是“两次握手”和“三次挥手”？

1. 为什么不能是“两次握手”？

TCP采用三次握手（而不是两次）主要为了解决以下核心问题：

问题 两次握手的缺陷 三次握手如何解决
历史连接干扰 若客户端旧的SYN报文延迟到达服务器，服务器直接分配资源，导致资源浪费。 客户端在第三次ACK中可拒绝无效连接（因序列号不匹配）。
双向序列号同步 两次握手只能确保客户端→服务器的序列号同步，服务器→客户端的序列号未被确认。 第三次握手明确确认了服务器的初始序列号（ACK=y+1），实现双向同步。
资源分配安全 服务器在第二次握手后立即分配资源，易受SYN洪泛攻击（伪造大量SYN耗尽服务器资源）。 服务器仅在收到第三次ACK后才分配资源，减少半连接占用。

示例场景：

  • 客户端发送SYN=1, SEQ=100（第一次握手）。
  • 服务器回复SYN=1, ACK=1, SEQ=300, ACK=101（第二次握手）。
  • 若只有两次握手：服务器认为连接已建立，但客户端可能因未收到响应而放弃连接，导致服务器资源长期占用。
  • 三次握手：客户端发送ACK=1, ACK=301（第三次握手）后，双方才确认连接有效。
2. 为什么不能是“三次挥手”？

TCP连接释放需四次挥手（而非三次），原因如下：

问题 三次挥手的缺陷 四次挥手如何解决
半关闭状态（Half-Close） 若一方发送FIN后立即关闭，可能丢失对方未发送完的数据。 允许单向连接关闭（如客户端FIN后仍可接收服务器数据）。
ACK与FIN的分离 FIN和ACK不能合并发送（因FIN是应用层触发，ACK是TCP层响应，时序可能不同）。 分开发送确保：
1. 先确认对方的FIN（ACK）。
2. 再发送自己的FIN（如有数据待发送）。
TIME_WAIT状态必要性 若跳过第四次ACK，最后一次ACK丢失会导致服务器持续重传FIN。 客户端发送第四次ACK后进入TIME_WAIT，确保服务器能收到最终确认。

示例场景：

  1. 客户端发送FIN=1, SEQ=200（第一次挥手）。
  2. 服务器回复ACK=1, ACK=201（第二次挥手）但继续发送剩余数据。
  3. 服务器数据发送完毕后，发送FIN=1, SEQ=500（第三次挥手）。
  4. 客户端回复ACK=1, ACK=501（第四次挥手）。

若只有三次挥手：

  • 服务器将第二次挥手（ACK）和第三次挥手（FIN）合并，可能延迟FIN的发送（需等待数据发送完毕），导致客户端长时间等待。

总结：

  • 握手：如同电话沟通，需双方确认“能听到吗？”（SYN）→“能听到”（SYN-ACK）→“好的，开始说话”（ACK）。
  • 挥手：如同挂电话，需双方明确“我说完了”（FIN）→“知道了”（ACK）→“我也说完了”（FIN）→“好的，挂断”（ACK）。

TCP可靠传输

机制 说明 示例
确认应答（ACK） 接收方对每个收到的数据段发送ACK（累积确认：ACK=N表示N之前所有数据已接收） 发送SEQ=100, Len=20 → 接收方回复ACK=120
超时重传 若发送方未收到ACK，在超时后重传数据（超时时间动态计算，如RTT加权平均） 发送数据后未收到ACK，等待200ms后重传
序列号排序 数据按序列号（SEQ）重组，解决乱序问题 接收SEQ=100和SEQ=120，先缓存SEQ=120直到SEQ=100到达
去重机制 通过序列号丢弃重复数据包 收到重复的SEQ=100时直接丢弃

TCP流量控制

概念 说明 数学关系
接收窗口（rwnd） 接收方通过ACK报文通告的剩余缓冲区大小（单位：字节） 发送窗口 = min(rwnd, cwnd)（同时受拥塞控制限制）
零窗口处理 若rwnd=0，发送方暂停发送并启动零窗口探测定时器（ZWP，默认每5-60s探测） 探测报文携带1字节数据，若rwnd>0则恢复传输
糊涂窗口综合征 避免发送微小数据段（如1字节），通过Nagle算法合并小包或延迟发送 条件：数据量≥MSS 或 收到前一个数据的ACK

TCP拥塞控制

阶段/算法 规则 窗口变化公式 目的
慢启动 每收到一个ACK，cwnd指数增长（初始cwnd=1 MSS） cwnd += MSS（每RTT翻倍） 快速探测可用带宽
拥塞避免 cwnd超过慢启动阈值（ssthresh）后，转为线性增长 cwnd += MSS*(MSS/cwnd)（每RTT增加1 MSS） 避免激进增长导致丢包
快重传 收到3个重复ACK时立即重传丢失报文（无需等待超时） ssthresh = cwnd/2, cwnd = ssthresh + 3 MSS 减少超时等待时间
快恢复 重传后直接进入拥塞避免阶段（而非慢启动） cwnd = ssthresh（之后线性增长） 避免带宽断崖式下降

注意：
超时的时候cwnd被置为1：（因为超时的时候，一个ACK都没有传过来，网络拥塞很严重，所以发送方得最大限度地一直数据发送量）
三个冗余ACK的时候cwnd减半：（因为收到三个冗余ACK的时候，代表有三个ACK报文段能被正确交付，网络拥塞没有那么严重，所以减半）

可靠传输是基础（解决端到端问题），流量控制解决接收方瓶颈，拥塞控制解决网络瓶颈。

TCP和UDP的比较

UDP TCP
是否连接 无连接 面向连接
是否可靠 不可靠传输，不使用流量控制和拥塞控制 可靠传输，使用流量控制和拥塞控制
连接对象个数 支持一对一，一对多，多对一和多对多交互通信 只能是一对一通信
传输方式 面向报文 面向字节流
首部开销 首部开销小，仅8字节 首部最小20字节，最大60字节
适用场景 适用于实时应用（IP电话、视频会议、直播等） 适用于要求可靠传输的应用，例如文件传输
  • TCP向上层提供面向连接的可靠服务 ，UDP向上层提供无连接不可靠服务。
  • 虽然 UDP 并没有 TCP 传输来的准确，但是也能在很多实时性要求高的地方有所作为
  • 对数据准确性要求高，速度可以相对较慢的，可以选用TCP
