状态模式

需求分析

假设我们有一个电梯系统，电梯有不同的状态（例如停止、运行、开门）。根据当前状态，电梯的行为（如运行、停止、开门）不同。

问题分析：

  1. 条件判断繁琐： 每次方法执行时，都需要通过判断 state 来决定执行哪一块代码，随着状态种类增加，条件判断的复杂度也随之增加。
  2. 维护困难： 如果电梯系统的状态越来越多，代码中的条件判断就会变得越来越复杂，维护和扩展变得非常困难，增加了出错的可能性。
  3. 状态与行为混杂： 电梯的状态和电梯的行为被紧密地耦合在一起，这违背了单一职责原则（SRP），一个类承担了过多的责任。

状态模式定义

状态模式是一种行为型设计模式，它允许一个对象在其内部状态改变时改变其行为。对象看起来好像修改了其类。状态模式通过将不同的状态封装到单独的类中，使得对象的状态变化能够独立于客户端进行管理，从而避免了复杂的状态判断逻辑。

有限状态机

状态模式与有限状态机 的概念紧密相关。

其主要思想是程序在任意时刻仅可处于几种有限的状态中。 在任何一个特定状态中， 程序的行为都不相同， 且可瞬间从一个状态切换到另一个状态。 不过， 根据当前状态， 程序可能会切换到另外一种状态， 也可能会保持当前状态不变。 这些数量有限且预先定义的状态切换规则被称为转移。

状态机通常由众多条件运算符 （ if或 switch ） 实现， 可根据对象的当前状态选择相应的行为。 “状态” 通常只是对象中的一组成员变量值。 即使你之前从未听说过有限状态机， 你也很可能已经实现过状态模式。 正如我们前面的错误示例那样。

真实世界类比

智能手机的按键和开关会根据设备当前状态完成不同行为：

  • 当手机处于解锁状态时， 按下按键将执行各种功能。
  • 当手机处于锁定状态时， 按下任何按键都将解锁屏幕。
  • 当手机电量不足时， 按下任何按键都将显示充电页面。

状态模式结构

状态模式建议为对象的所有可能状态新建一个类， 然后将所有状态的对应行为抽取到这些类中。

原始对象被称为上下文 （context）， 它并不会自行实现所有行为， 而是会保存一个指向表示当前状态的状态对象的引用， 且将所有与状态相关的工作委派给该对象。

状态模式通常包括以下角色：

  • 上下文（Context）：持有一个状态对象的引用，并通过这个引用来调用当前状态的行为。
  • 抽象状态（State）：定义一个接口，用于封装与上下文的每个状态相关的行为。
  • 具体状态（ConcreteState）：实现抽象状态接口，定义在特定状态下对象的行为。

状态模式实现

不使用状态模式时，整个状态机的控制逻辑集中在一个类中，导致了复杂的条件判断和易于出错的代码，相比之下，使用状态模式后，状态的逻辑被分离到了不同的状态类中，Elevator 类不再需要关心每个状态的行为，而是委托给各个具体状态类，避免了条件判断的繁杂，并且每个状态类只负责自己的行为，符合单一职责原则。

状态模式优缺点

优点 缺点
1. 状态的行为被封装到不同的状态类中，符合单一职责原则，易于扩展和维护。 1. 状态数量多时，会导致系统中类的数量增多，增加系统复杂性。
2. 状态切换的逻辑集中管理，代码清晰，易于理解。 2. 每个状态都需要为所有操作实现方法，可能导致类的代码重复。
3. 可以动态地修改对象的行为，使得对象的行为依赖于其状态，避免了复杂的条件判断。 3. 增加了类的数量和系统的复杂度，需要针对每个可能的状态编写具体类。

使用场景

对象的行为随其状态改变而不同当一个对象的行为根据其状态的不同而有所不同，并且需要在运行时改变对象的状态时，适合使用状态模式。

  • 例如，电梯、音响播放器、网络连接等有不同工作状态的系统。

避免复杂的条件判断语句在代码中频繁使用大量的 if-else 或 switch，判断不同状态的行为时，使用状态模式可以大大简化代码结构。

  • 例如，电商系统中根据订单状态（如待付款、已发货、已收货等）做不同操作。
