缘起Binder

站在巨人的肩膀上：Binder系列—开篇 - Gityuan博客 | 袁辉辉的技术博客

Binder是什么？

Android的Binder是一种进程间通信（IPC）机制，是Android系统中用于实现跨进程通信的核心技术。它允许不同进程之间安全、高效地传递数据和调用方法，支撑了Android应用组件（如Activity、Service）和系统服务之间的交互。

既然是IPC机制。总所周知，在Linux平台下有着众多选择，比如管道，共享内存，套接字等。详见CPU虚拟化与进程 - The Peak Tower。而Android底层又基于Linux内核构建的。为什么不直接采用Linux现有的进程IPC方案呢？

这个问题的答案众说纷纭：

(46 封私信 / 99+ 条消息) 为什么 Android 要采用 Binder 作为 IPC 机制？ - 知乎

其中高赞回答 Gityuan ，总结来说就是以下几点：

角度 Binder 的优势 传统 Linux IPC 的劣势
性能 - 数据拷贝次数：1 次（仅次于共享内存）。- 线程池模型，减少线程创建和销毁开销。- 支持同步调用。 - 管道、消息队列、Socket 需要 2 次拷贝。- 共享内存虽无需拷贝，但需要复杂的同步机制。
稳定性 - 基于 C/S 架构，架构清晰，客户端与服务端独立，稳定性较好。 - 共享内存实现复杂，需处理并发同步问题，易出现死锁。
安全性 - 内核层验证 UID/PID，确保进程身份。- 权限控制策略（如 SELinux）更严格。- 建立私有通道。 - 无法可靠获取对方进程的 UID/PID。- 访问接入点开放，无保护措施。
语言层面 - 符合 面向对象 思想，将进程间通信转化为对象方法调用。- 更适合基于 Java 的 Android 系统。 - Linux 基于 C 语言（面向过程），传统 IPC 机制不符合面向对象思想。
公司战略 - Binder 基于开源的 OpenBinder，采用 Apache-2.0 协议，避免 GPL 的“传染性”。- 减少法律风险。 - Linux 内核受 GPL 协议保护，用户空间调用内核功能需遵循 GPL，限制商业化。

可以说：”无Binder不Android”。Binder在Android中的作用可以直接类比Linux下的进程间通信。想象一下没有IPC,Linux的世界会有多糟糕

Binder体系架构

在我看来，要想总体把握的Binder机制，需要从两个方面来看：IPC和RPC

IPC（进程间通信）机制 是指操作系统提供的，用于在不同进程之间传递数据或信号的机制。由于进程是操作系统中的独立执行单元，每个进程都有自己独立的内存空间，因此进程之间不能直接访问彼此的数据。IPC 机制就是为了解决这个问题而设计的。

IPC是Binder的基础功能，负责在不同进程之间传递数据。

IPC原理

总结来说：对于用户空间，不同进程之间彼此是不能共享的，而内核空间却是可共享的。Client进程向Server进程通信，利用进程间可共享的内核内存空间来完成底层通信工作的。

RPC原理

RPC是建立在IPC之上的更高层抽象，允许一个进程调用另一个进程中的方法，就像调用本地方法一样。

RPC（远程过程调用）机制 是一种分布式计算技术，允许一个程序调用另一个地址空间（通常是远程机器上的进程）中的函数或方法，就像调用本地函数一样。RPC 的目标是隐藏网络通信的复杂性，使开发者能够像编写本地代码一样编写分布式应用。

而在Android系统中则利用了RPC的基本思想，建立在IPC之上的更高层抽象，允许一个进程调用另一个进程中的方法，就像调用本地方法一样。

仔细观察就能看到两者在结构上存在一定的相似。

Binder机制的底层实现通过IPC和RPC的有机结合，提供了高效、安全、易用的跨进程通信能力：

  • IPC部分体现在内核驱动、内存映射、事务管理和安全机制。(主要在native层)
  • RPC部分体现在代理对象、AIDL、方法调用封装和同步调用。(主要在FrameWork层)

Binder原理

Binder通信采用C/S架构(客户端发起请求，服务端处理请求。)：

其中Binder 驱动负责管理客户端和服务端之间的通信通道，处理事务传递和线程池管理。ServiceManager负责注册和查找 Binder 服务，客户端通过它获取服务的 Binder 引用。客户端通过代理对象（BinderProxy）调用远程方法，服务端通过桩类（Stub）解析请求并执行方法。

Binder 是一个 RPC框架。也就是说通过 Binder：

  • 可以在 A 进程中访问 B 进程中定义的函数
  • 进程 B 中的这些等待着被远程调用的函数的集合，我们称其为 Binder 服务（Binder Service）
  • 进程 A 称之为 Binder 客户端（Binder Client），进程 B 称之为 Binder 服务端（Binder Server）
  • 通常，系统中的服务很多，我们需要一个管家来管理它们，服务管家（ServiceManager） 是 Android 系统启动时，启动的一个用于管理 Binder 服务（Binder Service） 的进程。通常，服务（Service） 需要事先注册到服务管家（ServiceManager），其他进程向服务管家（ServiceManager） 查询服务后才能使用服务。
  • Binder 的 RPC 能力通过 Binder 驱动实现

通常一个完整的 Binder 程序涉及 4 个流程：

  1. 在 Binder Server 端定义好服务
  2. 然后向 ServiceManager 注册服务
  3. 在 Binder Client 中向 ServiceManager 获取到服务
  4. 发起远程调用，调用 Binder Server 中定义好的服务
